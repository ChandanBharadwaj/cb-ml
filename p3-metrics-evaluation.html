Metrics & Evaluation

“Is it good?” is the hardest question in ML, and the one that separates people who ship from people who demo. This page covers the classic metrics — precision, recall, the decision threshold — and the modern problem of evaluating LLMs, where there’s often no single right answer. Play with the slider below to feel why accuracy is rarely the number that matters.

Where this sits on the ladder
The judge for every rung. You can’t decide whether classical ML “is good enough” (and you can stop climbing) without measuring it honestly. Picking the right metric and threshold — not just training a model — is what turns a prototype into a decision you can defend to a customer.

Confusion matrix & the four numbers

Every binary classification outcome is one of four cells. Learn these cold — every metric is built from them:

Predicted +Predicted −
Actual +TP ✓ caught itFN ✗ missed it
Actual −FP ✗ false alarmTN ✓ correct pass
Why accuracy lies
If 1% of transactions are fraud, a model that predicts “never fraud” is 99% accurate and utterly useless. On imbalanced problems (fraud, disease, defects) accuracy is a trap — you need precision and recall.

Precision, recall, F1

  • Precision = TP / (TP + FP) — of the things I flagged, how many were right? Cost of a false alarm.
  • Recall = TP / (TP + FN) — of the things I should have caught, how many did I? Cost of a miss.
  • F1 = harmonic mean of the two — one number when you need a balance.
Which one you optimize IS a business decision
Spam filter → precision (don’t trash a real email). Cancer screening → recall (never miss a case; a false alarm just means another test). Fraud → depends on the cost of a blocked-legit vs a missed-fraud. You can’t pick the metric without the business context — and interviewers love this.

The threshold is a choice — drag it

A classifier outputs a score, not a decision. You pick the threshold above which a score counts as positive — and that choice trades precision against recall. Drag the slider: watch the same model give very different precision/recall as you move the line.

TP 0 FP 0 FN 0 precision 0 recall 0 F1 0
Points right of the line are predicted positive. Move it left → recall rises but precision falls (more false alarms). Move it right → precision rises but you miss more. Same model, your call.

ROC/AUC vs PR curves

Rather than commit to one threshold, plot performance across all of them:

  • ROC curve — true-positive rate vs false-positive rate across thresholds. AUC (area under it) = probability the model ranks a random positive above a random negative. 0.5 = random, 1.0 = perfect. Threshold-independent single number.
  • PR curve — precision vs recall. Preferred for imbalanced data, because ROC/AUC can look deceptively good when negatives vastly outnumber positives.

Calibration & class imbalance

A model is calibrated if its predicted probabilities mean what they say: among samples it scores 0.8, about 80% are truly positive. Ranking (AUC) can be great while calibration is terrible — and if you act on the probability (expected-value decisions, thresholds tied to cost), calibration matters. Fixes: Platt scaling, isotonic regression, temperature scaling.

For imbalance: use PR metrics, resample (SMOTE/undersample), class weights, and pick the threshold from the cost of each error — not the default 0.5.

Evaluating LLMs — when there’s no single right answer

Classification has a ground truth; “write a summary” doesn’t. Modern LLM eval is a toolbox:

MethodMeasuresNote
PerplexityHow well the model predicts held-out textIntrinsic; good for pretraining, weak for task quality
BLEU / ROUGEN-gram overlap with a referenceTranslation/summarization; brittle, misses paraphrase
BERTScoreEmbedding similarity to a referenceCaptures meaning better than n-grams
Exact match / unit testsIs the answer exactly right / does code pass?Great when there IS a right answer (math, code)
LLM-as-a-judgeA strong LLM scores outputs on a rubricScalable, flexible; the modern default for open-ended
Human evalPeople rate/rank outputsGold standard; slow and costly

Benchmarks you should recognize: MMLU (broad knowledge), HumanEval (code), GSM8K (grade-school math reasoning). For RAG specifically, measure faithfulness (grounded, no hallucination) and answer relevance (see RAG). Red-teaming — adversarially probing for unsafe or jailbreakable behavior — is the safety-side of evaluation.

FDE reality
Public benchmarks are contaminated and rarely match your task. The thing that actually ships is a small, task-specific eval set of real examples with an LLM-as-judge or human review — your own regression test suite for prompts, RAG, and models. Build that before optimizing anything.

Interview Q&A

corePrecision vs recall — when do you optimize which?

Precision (of what I flagged, how much was right) matters when false positives are costly — spam filtering, blocking legitimate transactions. Recall (of what I should catch, how much I caught) matters when misses are costly — cancer screening, fraud, safety. You choose based on the relative cost of a false alarm vs a miss, which is a business decision, and F1 balances the two when you need a single number. There’s no universally “right” one — it depends on the problem.

trapYour model is 99% accurate. Are you happy?

Not without knowing the base rate. If positives are 1% of the data, predicting “always negative” scores 99% and catches nothing — accuracy is meaningless on imbalanced problems. I’d look at precision, recall, the PR curve, and the confusion matrix, and choose a threshold from the cost of each error type. High accuracy on skewed data is a red flag, not a win.

coreWhat does the decision threshold do, and how do you pick it?

The model outputs a score; the threshold converts it to a yes/no. Lowering it increases recall but decreases precision (more false alarms); raising it does the opposite. You pick it by mapping each error to its business cost and choosing the operating point that minimizes total cost (or hits a required recall/precision) — often via the PR curve — rather than defaulting to 0.5. It’s a deliberate choice, not a fixed property of the model.

seniorROC/AUC or PR curve — which for a rare-positive problem?

PR. ROC/AUC uses the false-positive rate, which stays low simply because negatives are so numerous, so AUC can look great even when the model floods you with false positives relative to the few true positives. The precision-recall curve focuses on the positive class and exposes that tradeoff directly, so it’s far more informative under heavy class imbalance.

seniorHow do you evaluate an LLM feature where there’s no single correct answer?

Build a task-specific eval set of representative real inputs with either reference answers or a scoring rubric, then measure with the right tool: exact-match/unit-tests where a right answer exists (math, code, extraction), embedding or LLM-as-judge scoring for open-ended text, and human review as the gold standard on a sample. For RAG, score faithfulness and answer relevance separately. Treat it as a regression suite you run on every prompt/model change — public benchmarks are a starting signal, not proof it works on your task.

trapWhat’s the risk of using an LLM as a judge?

LLM judges have biases: they can favor longer answers, prefer their own model family, be sensitive to option order, and be gamed by confident-sounding but wrong text. Mitigate with clear rubrics, pairwise comparisons, randomized positions, a strong judge model, and periodic calibration against human labels. It’s scalable and often good enough, but you validate the judge itself rather than trusting it blindly.

Code: metrics + a threshold sweep, and LLM-as-judge

The threshold sweep is exactly what the slider above does, in code — compute precision/recall at every threshold and pick the operating point your business needs.

pythonconfusion matrix + threshold sweep (scikit-learn)
# Metrics from scores, and choosing a threshold by business cost (not 0.5).
# pip install scikit-learn numpy
import numpy as np
from sklearn.metrics import (confusion_matrix, precision_recall_curve,
                             roc_auc_score, average_precision_score)

y_true  = np.array([0, 0, 1, 1, 0, 1, 0, 1, 1, 0])          # ground truth
y_score = np.array([.1, .4, .35, .8, .2, .65, .5, .9, .55, .3])  # model scores

# Threshold-independent quality:
print("ROC-AUC:", round(roc_auc_score(y_true, y_score), 3))         # ranking quality
print("PR-AUC :", round(average_precision_score(y_true, y_score), 3))  # better under imbalance

# Sweep every threshold -> precision/recall pairs (what the slider visualizes).
prec, rec, thr = precision_recall_curve(y_true, y_score)

# Pick the threshold that meets a business rule, e.g. "recall must be >= 0.8".
ok = rec[:-1] >= 0.80                     # last point has no threshold; drop it
best_t = thr[ok][np.argmax(prec[:-1][ok])]  # among those, maximize precision
print("chosen threshold:", round(float(best_t), 2))

# Apply it and inspect the confusion matrix at THAT operating point.
y_pred = (y_score >= best_t).astype(int)
tn, fp, fn, tp = confusion_matrix(y_true, y_pred).ravel()
print(f"TP={tp} FP={fp} FN={fn} TN={tn}")
# The threshold is a dial you set from costs — not a fixed property of the model.
pythona minimal LLM-as-judge eval harness
# Evaluate open-ended outputs with an LLM judge scoring against a rubric.
# pip install openai pydantic
from openai import OpenAI
from pydantic import BaseModel
client = OpenAI()

class Judgement(BaseModel):
    score: int          # 1-5
    reason: str

JUDGE_RUBRIC = (
    "You are a strict evaluator. Score the ANSWER's faithfulness to the CONTEXT "
    "from 1 (fabricated) to 5 (fully grounded). Return JSON {score, reason}."
)

def judge(context: str, answer: str) -> Judgement:
    resp = client.chat.completions.create(
        model="gpt-4o",                       # use a STRONG model as the judge
        temperature=0,                        # deterministic scoring
        response_format={"type": "json_object"},
        messages=[
            {"role": "system", "content": JUDGE_RUBRIC},
            {"role": "user", "content": f"CONTEXT:\n{context}\n\nANSWER:\n{answer}"},
        ],
    )
    return Judgement.model_validate_json(resp.choices[0].message.content)

# Run across your eval set and aggregate — this is your regression suite.
evalset = [
    {"context": "The Pro plan costs $49/mo.", "answer": "It's $49 a month."},   # grounded
    {"context": "The Pro plan costs $49/mo.", "answer": "It's $99 a month."},   # wrong
]
scores = [judge(e["context"], e["answer"]).score for e in evalset]
print("mean faithfulness:", sum(scores) / len(scores))
# Validate the judge against a few human labels before trusting it at scale.
The habit to build
Whatever the rung — GBM or LLM — define the metric and the eval set before you optimize. The metric is your business goal in numbers; the threshold is your cost tradeoff made explicit. Models are easy; measuring them honestly is the job.
