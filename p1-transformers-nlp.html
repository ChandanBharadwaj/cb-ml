Transformers & NLP

The architecture behind every modern LLM. If an interviewer goes deep on one thing, it is attention — so this page derives it from scratch. By the end you can explain why transformers replaced RNNs, draw self-attention, and defend the design choices.

Where this sits on the ladder
Transformers are rung 3–4. They shine on sequence / language data where order and long-range context matter and hand-written rules or tabular classical ML can’t capture meaning. They are not the answer for a 10-column spreadsheet — that’s still gradient-boosted trees on a CPU.

Why transformers replaced RNNs

RNNs/LSTMs read a sequence one token at a time, carrying a hidden state forward. Two problems: (1) they can’t parallelize across the sequence — token t needs token t-1’s state, so training is slow; (2) long-range dependencies fade as the signal is squeezed through many steps (vanishing gradients).

The 2017 paper “Attention Is All You Need” threw out recurrence. Every token looks at every other token directly, in parallel, through self-attention. That single change unlocked GPU-parallel training on internet-scale data — which is what made LLMs possible.

One-line answer
“Transformers replace sequential recurrence with parallel self-attention, so every token can attend to every other token in O(1) path length — enabling long-range context and massively parallel training.”

Tokenization (BPE)

Models don’t see text; they see integer token IDs. Modern LLMs use subword tokenization (Byte-Pair Encoding / WordPiece / SentencePiece): common words become a single token, rare words split into reusable pieces. This keeps the vocabulary small (~30k–100k) while still representing any string — including typos and new words.

“tokenization” is rare, so it splits into token+iza+tion; common words stay whole. Each token maps to an integer ID.
FDE gotcha
Tokens ≠ words. English averages ~1.3 tokens/word; code and non-Latin scripts cost more. Since you pay per token and context windows are measured in tokens, a “4k word” document can blow a budget. Always estimate tokens, not words.

Embeddings & positional encoding

Each token ID indexes into an embedding table — a learned vector (e.g. 768–4096 dims) that places the token in a semantic space where similar meanings sit close together. But attention is permutation-invariant: on its own it can’t tell “dog bites man” from “man bites dog.” So we inject position.

The original transformer added fixed sinusoidal position signals. Modern LLMs (LLaMA, GPT-NeoX, most 2023+) use RoPE (Rotary Position Embedding), which rotates the query/key vectors by an angle proportional to position — so attention naturally depends on relative distance and extrapolates to longer contexts.

Position is encoded as a bank of sinusoids at different frequencies. Every position gets a unique fingerprint; nearby positions get similar ones.

Self-attention, derived

This is the heart of it. For each token we compute three vectors by multiplying its embedding by learned matrices:

  • Query (Q) — “what am I looking for?”
  • Key (K) — “what do I offer?”
  • Value (V) — “what I’ll actually pass on if attended to.”

A token’s new representation is a weighted average of every token’s Value, where the weight is how well this token’s Query matches that token’s Key. The full formula:

Scaled dot-product attention
Attention(Q, K, V) = softmax( Q·Kᵀ / √dk ) · V
Every token builds its meaning from a weighted blend of the others. Here “sat” leans on its subject “cat” and object “mat.”

Why divide by √dk?

A classic interview follow-up. The dot product Q·Kᵀ sums over dk dimensions, so its variance grows with dk. Large values push softmax into saturated regions where gradients vanish and it behaves like a hard argmax. Dividing by √dk keeps the scores unit-variance, so the softmax stays smooth and trainable.

Show the causal-mask detail (decoders)

In a decoder (GPT-style, generating left-to-right) a token must not see the future. Before the softmax we add a causal mask that sets all “future” scores to -∞, so their softmax weight becomes 0. That’s the only difference that makes attention autoregressive.

Multi-head attention

One attention pattern isn’t enough — a token may need syntactic and semantic relationships at once. So we run attention h times in parallel (e.g. 8–96 heads), each with its own small Q/K/V projections, then concatenate and project back. Each head can specialize: one tracks subject→verb, another tracks coreference, etc.

Split into h heads → each attends independently → concatenate → linear projection. Same total compute, far richer representations.

The transformer block

A block stacks two sub-layers, each wrapped in a residual connection and layer normalization: (1) multi-head self-attention, (2) a position-wise feed-forward network (FFN). Stack N of these (GPT-3: 96) and you have the model.

Residual connections (green) let gradients flow straight through, so you can stack dozens of blocks without the signal degrading.

Encoder vs decoder vs encoder-decoder

TypeAttentionGood atExamples
Encoder-onlyBidirectional (sees full context)Understanding: classification, embeddings, retrievalBERT, sentence-transformers
Decoder-onlyCausal (left-to-right)Generation: chat, completion, codeGPT, LLaMA, Mistral
Encoder-decoderEncoder bidirectional + decoder cross-attendsSeq-to-seq: translation, summarizationT5, BART
Rule of thumb
Need a vector that represents text (search, RAG, classification)? Reach for an encoder. Need to produce text? Reach for a decoder. Almost every chat LLM today is decoder-only.

Interview Q&A

coreWalk me through self-attention.

Project each token embedding into Query, Key and Value vectors. Score every token pair with the dot product of one token’s Query and another’s Key, scale by 1/√dk, softmax the scores into weights, then take the weighted sum of Values. That weighted sum is the token’s new, context-aware representation. It’s done for all tokens at once as matrix multiplications, which is why it’s GPU-friendly.

trapWhy the √dk scaling — what breaks without it?

Without scaling, dot products grow with dimension (variance ∝ dk). Large logits saturate the softmax → near-zero gradients → the model can’t learn, and attention collapses to a hard pick of one token. Dividing by √dk normalizes the score variance to ~1, keeping softmax in its sensitive, differentiable range.

coreWhat’s the computational complexity of self-attention, and why does it matter?

O(n²·d) in sequence length n — every token attends to every other, so the attention matrix is n×n. That quadratic cost is exactly why long context is expensive and why techniques like FlashAttention, sliding-window attention, and KV-caching exist. (See the Inference Optimization page.)

seniorWhy do modern LLMs use RoPE instead of the original sinusoidal encodings?

Sinusoidal/learned absolute encodings are added to the input once. RoPE instead rotates Q and K by an angle tied to position, so the attention score between two tokens depends on their relative offset. That generalizes better to positions unseen in training and makes context-length extension (e.g. NTK/YaRN scaling) tractable — which is why almost all 2023+ open models adopted it.

trapIf attention is permutation-invariant, how does the model know word order?

It doesn’t — until you add positional information. Attention treats the input as a set. Positional encodings (sinusoidal, learned, or RoPE) are what inject order, so the model can distinguish “dog bites man” from “man bites dog.”

seniorEncoder-only vs decoder-only — which would you pick for semantic search, and why?

Encoder-only (e.g. a sentence-transformer). Search needs a single, high-quality embedding per document, and bidirectional attention lets every token see the full context, producing better representations. Decoder-only models are optimized to generate the next token, not to summarize meaning into one vector — though you can pool their hidden states, encoders are usually cheaper and stronger for pure retrieval.

Code: attention from scratch, then with Hugging Face

First the mechanics in NumPy — no framework magic, so you can see every step. This is the exact thing an interviewer may ask you to whiteboard.

pythonscaled dot-product self-attention (NumPy)
# Scaled dot-product self-attention, implemented from scratch to show the mechanics.
# No deep-learning framework — just NumPy — so every step is visible.
import numpy as np

def softmax(x, axis=-1):
    # Subtract the max for numerical stability (avoids overflow in exp).
    x = x - x.max(axis=axis, keepdims=True)
    e = np.exp(x)
    return e / e.sum(axis=axis, keepdims=True)

def self_attention(X, W_q, W_k, W_v, causal=False):
    """
    X   : (seq_len, d_model)  input token embeddings (one row per token)
    W_* : (d_model, d_k)      learned projection matrices for Q, K, V
    Returns: (seq_len, d_k)   context-aware representation for each token
    """
    # 1) Project each token into Query, Key, Value spaces.
    Q = X @ W_q                      # (seq_len, d_k) — "what each token is looking for"
    K = X @ W_k                      # (seq_len, d_k) — "what each token offers"
    V = X @ W_v                      # (seq_len, d_k) — "what each token passes on"

    d_k = Q.shape[-1]

    # 2) Score every query against every key: how well do they match?
    scores = Q @ K.T                 # (seq_len, seq_len)

    # 3) Scale by sqrt(d_k) so the softmax doesn't saturate (the key trick).
    scores = scores / np.sqrt(d_k)

    # 4) Optional causal mask: a token must not attend to future tokens (decoders).
    if causal:
        seq_len = scores.shape[0]
        # Upper-triangular positions (above the diagonal) are "the future".
        mask = np.triu(np.ones((seq_len, seq_len)), k=1).astype(bool)
        scores[mask] = -np.inf       # -inf -> softmax weight becomes exactly 0

    # 5) Turn scores into probabilities (each row sums to 1).
    weights = softmax(scores, axis=-1)   # (seq_len, seq_len)

    # 6) Weighted sum of Values = the new representation of each token.
    return weights @ V, weights          # output (seq_len, d_k), plus weights to inspect

# --- tiny demo ---------------------------------------------------------------
np.random.seed(0)
X = np.random.randn(4, 8)            # 4 tokens, 8-dim embeddings
Wq, Wk, Wv = (np.random.randn(8, 8) for _ in range(3))
out, attn = self_attention(X, Wq, Wk, Wv, causal=True)
print("output shape:", out.shape)   # (4, 8)
print("attention row 0 (sums to 1):", attn[0].round(3))

In practice you never hand-roll this — you use a library. Here’s the same idea at the level you actually work at, plus getting real embeddings out of a pretrained model:

pythontokenizer + embeddings with Hugging Face
# The real-world version: tokenize text and pull embeddings from a pretrained model.
# pip install transformers torch
from transformers import AutoTokenizer, AutoModel
import torch

model_name = "sentence-transformers/all-MiniLM-L6-v2"   # a small encoder — CPU-friendly
tokenizer = AutoTokenizer.from_pretrained(model_name)
model = AutoModel.from_pretrained(model_name)

sentences = ["The cat sat on the mat.", "A feline rested on a rug."]

# 1) Tokenize: text -> token IDs (+ attention mask for padding).
enc = tokenizer(sentences, padding=True, truncation=True, return_tensors="pt")
print("token IDs:\n", enc["input_ids"])

# 2) Forward pass through the transformer (no gradients needed for inference).
with torch.no_grad():
    out = model(**enc)                       # last_hidden_state: (batch, seq_len, hidden)

# 3) Mean-pool token vectors into one sentence embedding (mask out padding).
mask = enc["attention_mask"].unsqueeze(-1)   # (batch, seq_len, 1)
summed = (out.last_hidden_state * mask).sum(1)
counts = mask.sum(1).clamp(min=1e-9)
embeddings = summed / counts                 # (batch, hidden) — one vector per sentence

# 4) Cosine similarity: the two paraphrases should score high despite different words.
a, b = torch.nn.functional.normalize(embeddings, dim=1)
print("cosine similarity:", float(a @ b))    # ~0.7+ — semantic match, not word overlap
Connect it to what you know
That sentence embedding is just a feature vector — the same thing you’d compute in a Spark/ Databricks pipeline, except the “feature engineering” is a transformer. Store those vectors in a vector DB and you have semantic search (see Vector DBs & Embeddings).
