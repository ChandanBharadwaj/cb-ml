Inference Optimization

A model that’s accurate but too slow or too expensive doesn’t ship. This page is the gap between “it works in a notebook” and “it serves 1,000 users under a 2-second SLA” — the KV cache, quantization, batching, and serving tricks that make LLMs fast and cheap, plus the real-time thresholds you’re actually measured against.

Where this sits on the ladder
The operations layer of rung 4. None of this changes what the model knows — it changes what it costs to run. It’s also the strongest argument for staying on a lower rung: a GBM answers in microseconds on a CPU, while an LLM needs all these tricks just to hit a sane latency. Optimization is how you make rung 4 affordable when you truly need it.

The metrics: TTFT, TPOT, throughput

You can’t optimize what you don’t name. LLM serving has its own vocabulary of thresholds:

MetricWhatWhy it matters
TTFT
(time to first token)		Latency until the first token appearsPerceived responsiveness in a chat UI
TPOT / ITL
(time per output token)		Gap between subsequent tokensHow fast the stream “types”
ThroughputTotal tokens/sec across all requestsCapacity & cost efficiency (tokens per $)
p50 / p95 / p99Latency percentiles, not averagesSLAs live in the tail — p95 is what users feel
FDE framing
There’s a fundamental latency ↔ throughput ↔ cost triangle. Bigger batches raise throughput (cheaper per token) but hurt per-request latency; smaller models and quantization help both at some quality cost. You don’t “make it fast” — you pick an operating point that meets the p95 SLA at acceptable cost. Always design to percentiles, never averages.

Prefill vs decode & the KV cache

LLM inference has two phases with very different cost profiles:

  • Prefill — process the whole prompt in parallel to produce the first token. Compute-bound; fast per token, done once.
  • Decode — generate the rest one token at a time. Memory-bandwidth-bound; this is where most of the latency lives, and it can’t be parallelized across the sequence.

The KV cache is the key optimization: instead of recomputing attention over all previous tokens at every step, the model caches the keys and values of past tokens and reuses them. It turns per-step cost from “re-read the whole sequence” into “process one new token.” The catch: the cache grows with sequence length × layers × heads and can dominate GPU memory on long contexts — which is exactly what the next tricks attack.

Each generated token appends a key/value entry. Longer sequences (and more concurrent requests) mean a bigger cache — often the real memory bottleneck, not the model weights.

GQA & MoE — cheaper by design

  • GQA (Grouped-Query Attention) — multiple query heads share one set of key/value heads. That shrinks the KV cache several-fold with minimal quality loss, so it’s in nearly every modern model. (MQA is the extreme: all queries share one KV head.)
  • MoE (Mixture of Experts) — the feed-forward layer is split into many “expert” sub-networks, and a router activates only a few per token. So a model can have huge total capacity (e.g. hundreds of billions of params) while only using a fraction per token — big quality, lower per-token compute. The tradeoff: all experts must sit in memory.
  • FlashAttention — an exact but IO-aware attention kernel that avoids writing the big attention matrix to slow memory. Same math, much faster and lower memory. Free win.

Quantization

Store and compute weights (and sometimes the KV cache) in lower precision to cut memory and speed things up. The common ladder: FP16/BF16 (standard) → INT8 (~2× smaller, tiny quality loss) → INT4 (~4× smaller, more loss). Because LLM decoding is memory-bandwidth-bound, moving fewer bytes per token often makes inference faster too, not just smaller. Methods you can name: GPTQ, AWQ, bitsandbytes (NF4).

Rule of thumb
INT8 is nearly free quality-wise and almost always worth it for serving. INT4 is the aggressive option when memory/cost is tight — test quality on your task first. Quantizing the KV cache (not just weights) is an increasingly common way to fit longer contexts.

Batching, vLLM & speculative decoding

  • Continuous (in-flight) batching — instead of waiting to batch requests together, the server swaps new requests in as others finish each token. Massively higher GPU utilization and throughput. This is the single biggest serving win.
  • vLLM & PagedAttention — vLLM manages the KV cache like OS virtual memory, in non-contiguous “pages,” eliminating the memory fragmentation that wasted capacity. It’s the de-facto high-throughput serving engine (SGLang and TensorRT-LLM are peers).
  • Speculative decoding — a small “draft” model proposes several tokens, and the big model verifies them in one parallel pass, accepting the correct prefix. Same output distribution, fewer big-model steps → lower latency. A clever way to beat the sequential-decode bottleneck.

Caching

  • Prompt / prefix caching — reuse the KV cache for a shared prefix (a long system prompt, a fixed RAG context) across requests, so you don’t re-run prefill every time. Big TTFT and cost win for repeated prompts.
  • Semantic caching — if a new query is embedding-similar to a past one, return the cached answer instead of calling the model. Cuts cost and latency for common/duplicate questions — but you must bound similarity carefully to avoid wrong hits.

Interview Q&A

coreWhat is the KV cache and why does it matter?

During generation the model would otherwise recompute attention keys and values for every prior token at each step. The KV cache stores those keys/values and reuses them, so each new step only processes the single new token — turning quadratic re-computation into linear work. The cost is memory: the cache grows with sequence length × layers × KV-heads and often becomes the dominant memory consumer, which is why GQA, paged attention, and KV-cache quantization exist.

coreWhich metrics define an LLM serving SLA?

Time to first token (TTFT) for perceived responsiveness, time per output token (TPOT/ITL) for streaming speed, and throughput (tokens/sec) for capacity and cost. Crucially you track percentiles — p95/p99 — not averages, because the tail is what users actually experience. And you balance the latency–throughput–cost triangle: batching improves throughput/cost but can hurt per-request latency.

seniorWhy is continuous batching such a big deal?

Requests finish at different times, so static batching wastes the GPU waiting for the slowest one and can’t admit new work mid-flight. Continuous (in-flight) batching schedules at the token level — as soon as one sequence emits a token or finishes, another request’s tokens fill the slot — keeping the GPU saturated. It can multiply throughput several-fold with the same hardware, which is why engines like vLLM are built around it.

seniorHow does speculative decoding speed up generation without changing the output?

A small, cheap draft model proposes the next few tokens; the large target model then verifies all of them in a single parallel forward pass and accepts the longest correct prefix (rejecting from the first mismatch). Because verification is parallel and cheaper than generating each token sequentially with the big model, you get fewer expensive steps for the same result. With a proper acceptance rule the output distribution is identical to plain decoding — it’s a latency optimization, not a quality tradeoff.

trapYour average latency is fine but users complain it’s slow. What do you check?

Averages hide the tail — look at p95/p99. A good mean with a bad p99 means some requests are far too slow (long inputs, cache misses, queueing under load, cold starts, GC/allocator stalls). Also separate TTFT from total time: if TTFT is high the UI feels frozen even if total throughput is fine. Fix by designing to percentiles: cap context length, add prefix caching, tune batch size, autoscale, and shed or queue load gracefully.

coreWhen is quantization worth it, and what’s the risk?

Almost always worth it for serving: INT8 roughly halves memory with negligible quality loss, and because decode is memory-bandwidth-bound, moving fewer bytes can also speed it up. INT4 saves more but with a larger quality hit, so validate on your task. The risk is silent quality degradation on edge cases, so you quantize and re-run your evals rather than trusting that “it still sounds fine.”

Code: measuring latency & loading quantized

You can’t hit an SLA you don’t measure. First, the two numbers that matter — TTFT and total time — via streaming; then loading a model in 4-bit to cut memory.

pythonmeasure TTFT + tokens/sec by streaming
# Measure the metrics that define your SLA: TTFT and output tokens/sec.
# pip install openai
from openai import OpenAI
import time

client = OpenAI()
start = time.perf_counter()
first_token_at = None
n_tokens = 0

# stream=True lets us time the FIRST token separately from the rest.
stream = client.chat.completions.create(
    model="gpt-4o-mini",
    messages=[{"role": "user", "content": "Explain KV cache in 3 sentences."}],
    stream=True,
)
for chunk in stream:
    delta = chunk.choices[0].delta.content
    if delta:
        if first_token_at is None:
            first_token_at = time.perf_counter()      # <- TTFT boundary
        n_tokens += 1

end = time.perf_counter()
ttft = first_token_at - start                          # time to first token (perceived latency)
gen_time = end - first_token_at                        # time spent streaming the rest
print(f"TTFT:        {ttft*1000:.0f} ms")
print(f"output tok/s: {n_tokens / gen_time:.1f}")      # streaming speed (TPOT ~ 1/this)
# In load tests you'd collect these across many requests and report p50/p95/p99,
# NOT the average — the SLA lives in the tail.
pythonload a model in 4-bit to cut GPU memory
# Self-hosting? Quantize on load to fit bigger models on smaller GPUs.
# pip install transformers bitsandbytes accelerate
from transformers import AutoModelForCausalLM, AutoTokenizer, BitsAndBytesConfig
import torch

# NF4 4-bit config: ~4x smaller weights than FP16, small quality cost.
bnb = BitsAndBytesConfig(
    load_in_4bit=True,
    bnb_4bit_quant_type="nf4",              # normalized float 4 — good default
    bnb_4bit_compute_dtype=torch.bfloat16,  # compute in bf16 for stability
    bnb_4bit_use_double_quant=True,         # extra compression of the quant constants
)

tok = AutoTokenizer.from_pretrained("Qwen/Qwen2.5-7B-Instruct")
model = AutoModelForCausalLM.from_pretrained(
    "Qwen/Qwen2.5-7B-Instruct",
    quantization_config=bnb,                # a 7B model now fits in ~5-6 GB instead of ~14 GB
    device_map="auto",
)
# Same API as full precision — you traded a little quality for a lot of memory.
# Always re-run your evals after quantizing; don't trust "it still sounds fine".
The pattern to remember
Optimization order for serving: quantize (fit it) → continuous batching + vLLM (utilize the GPU) → prefix/semantic caching (skip repeat work) → speculative decoding (shave latency) — all measured against p95, not averages.
