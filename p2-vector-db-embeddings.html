Vector DBs & Embeddings

The storage engine under RAG and semantic search. If embeddings turn meaning into geometry, a vector database is what lets you find the nearest points among millions in milliseconds. This is the piece that’s most like the data infrastructure you already run — with one new primitive: similarity instead of exact match.

Where this sits on the ladder
Infrastructure, not a model. A vector index is cheap (mostly CPU + RAM) and powers rung-2-to-4 features: semantic search, RAG retrieval, dedup, recommendations, clustering. It’s the enabling layer — the LLM is optional on top of it.

Embeddings: meaning as geometry

An embedding is a fixed-length vector (say 384–1536 numbers) that represents the meaning of text (or an image, or audio) as a point in high-dimensional space. The trick: the model is trained so that semantically similar things land near each other. “How do I reset my password?” sits close to “I forgot my login,” even with no shared words.

Similar meanings cluster. Search becomes “find the nearest points to the query’s embedding” — nearness = relevance.
Choosing an embedding model
Bigger isn’t always better. Pick by: domain fit (general vs code vs legal), dimensionality (more dims = more storage & slower search), context length (how big a chunk it can embed), and cost (local vs API). Check the MTEB leaderboard, but validate on your data — leaderboard rank rarely survives contact with a real corpus. Critically: the same model must embed both documents and queries.

The normalization shortcut
If you L2-normalize vectors (unit length), cosine similarity and dot product give the same ranking — and a dot product is cheaper. That’s why most pipelines normalize once at index time and use dot product thereafter. Mixing a cosine model with an L2 index is a classic silent bug.

Why ANN, not brute force

Comparing a query against every vector (exact / flat search) is O(n·d) — fine for thousands of vectors, hopeless for millions per request. Approximate Nearest Neighbor (ANN) indexes trade a tiny bit of recall for orders-of-magnitude speed: they get the right neighbors ~95–99% of the time in sub-millisecond time. That recall-vs-speed dial is the core tradeoff of every vector database.

HNSW vs IVF — the two you must know

HNSW greedily hops through a multi-layer “small-world” graph — a few big jumps up top, fine steps at the bottom — reaching the neighborhood in log-ish time.
IndexHowStrengths / tradeoffs
HNSWLayered navigable small-world graphExcellent recall + speed; higher memory; the common default
IVFCluster vectors, search only the nearest clusters (probes)Memory-efficient at huge scale; recall depends on #probes
FlatExact brute force100% recall; only for small collections / ground truth

Quantization & dimensionality

Millions of float32 vectors eat RAM fast (1M × 768 dims × 4 bytes ≈ 3 GB). Two levers shrink it:

  • Quantization — store vectors in lower precision. Scalar (float32→int8) ≈ 4× smaller with minor recall loss; Product Quantization (PQ) compresses far more for billion-scale, with a bigger recall hit.
  • Dimensionality — fewer dims = less storage and faster search. Some models (Matryoshka embeddings) let you truncate dimensions with graceful quality loss.

Metadata filtering & hybrid

Real retrieval is rarely “pure similarity.” You attach metadata to each vector (tenant, date, doc type, permissions) and filter on it — “nearest chunks from this customer, newer than Jan, that this user may read.” This is also how you enforce multi-tenancy and access control in a shared index — miss it and you leak one customer’s data into another’s results. Combine with keyword (sparse) scoring for hybrid search (see RAG).

FDE gotcha: filtered ANN is subtle
Applying a hard metadata filter to an approximate graph can wreck recall (the reachable neighbors get filtered out). Good vector DBs handle pre- vs post-filtering carefully — know that filtering and ANN interact, and test recall with your filters, not just without them.

Interview Q&A

coreWhat is a vector database and why not just use Postgres?

A vector database stores embeddings and answers nearest-neighbor queries efficiently via ANN indexes (HNSW/IVF). A regular DB is built for exact matches and range queries; scanning millions of vectors for similarity per request is too slow. That said, extensions like pgvector add ANN to Postgres and are perfect up to moderate scale — so “just use Postgres+pgvector” is often the right, cheaper answer before reaching for a dedicated system.

trapCosine vs dot product — does it matter?

Cosine measures angle only; dot product includes magnitude. If your vectors are L2-normalized, the two produce identical rankings, and dot product is cheaper — which is why pipelines normalize once and use dot product. The bug to avoid: using a metric that doesn’t match how the embedding model was trained (e.g. an L2 index with a cosine-trained model), which silently degrades recall.

seniorExplain the recall-vs-latency tradeoff in ANN indexes.

ANN is approximate on purpose: it skips most of the space to be fast, occasionally missing a true nearest neighbor. Index parameters control the dial — for HNSW, higher ef_search / M explore more of the graph for better recall at higher latency/memory; for IVF, more nprobe clusters means better recall but slower search. You tune these to hit a target recall (say 98%) within your latency SLA — there’s no free lunch, only the operating point you choose.

seniorHow do you do access control in a shared vector index?

Store authorization metadata (tenant id, allowed roles, document ACLs) alongside each vector and apply it as a filter at query time, so a user only ever retrieves chunks they’re permitted to see. For strict isolation you can also partition by namespace/collection per tenant. The risk to call out: naive filtering on an approximate index can hurt recall, so you must verify recall with the filters applied — and never rely on the LLM prompt to enforce permissions.

coreWhen would you quantize your vectors?

When memory/cost is the bottleneck. Scalar quantization (int8) cuts storage ~4× with small recall loss — a good default at scale. Product quantization compresses much further for billion-scale collections at a larger recall cost. Below a few million vectors it’s usually unnecessary — full precision is fine and simpler. Measure recall before and after; don’t quantize blindly.

Code: build & query a vector index

A real ANN index with FAISS — embed, add, search — plus the normalization detail that makes inner product behave as cosine. This is the engine you’d put behind the RAG retrieve step.

pythonbuild an HNSW index and query it (FAISS)
# A production-shaped vector index with FAISS.
# pip install faiss-cpu sentence-transformers numpy
import faiss
import numpy as np
from sentence_transformers import SentenceTransformer

docs = [
    "How do I reset my password?",
    "Our refund policy lasts 30 days.",
    "Login fails with a 403 error.",
    "The Pro plan includes 10 seats.",
]

model = SentenceTransformer("all-MiniLM-L6-v2")
# normalize_embeddings=True -> unit vectors, so INNER PRODUCT == cosine similarity.
vecs = model.encode(docs, normalize_embeddings=True).astype("float32")
dim = vecs.shape[1]                       # 384 for this model

# --- Build an HNSW index (great recall/speed default) -----------------------
# 32 = graph connectivity M; higher = better recall, more memory.
index = faiss.IndexHNSWFlat(dim, 32, faiss.METRIC_INNER_PRODUCT)
index.hnsw.efConstruction = 200           # build-time thoroughness
index.hnsw.efSearch = 64                  # query-time recall/latency dial
index.add(vecs)                           # index now holds all document vectors

# --- Query ------------------------------------------------------------------
def search(query: str, k: int = 2):
    q = model.encode([query], normalize_embeddings=True).astype("float32")  # SAME model!
    scores, ids = index.search(q, k)      # returns (similarities, indices)
    return [(docs[i], float(s)) for i, s in zip(ids[0], scores[0])]

for doc, score in search("I can't log in"):
    print(f"{score:.3f}  {doc}")
# Top hit is the 403/login doc — matched by MEANING, not shared keywords.

# In production you'd add metadata + filtering, persist the index, and (at scale)
# quantize the vectors — but the retrieve primitive is exactly this.
Connect it to what you know
Think of the vector index as a specialized read replica: an offline job (Spark/Databricks) computes embeddings and bulk-loads them; the serving path issues low-latency similarity queries. Same build-vs-serve split you already run — the only new idea is “nearest” replacing “equals.”
