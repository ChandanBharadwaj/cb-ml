Problem → Solution

The FDE’s core skill, and how senior interviews are framed: given a messy business problem, scope it — why won’t a plain algorithm do, which model (or combination) fits, and how do the pieces connect end to end. This page is a set of worked case studies you can pattern-match against, each following the same shape: problem → why plain code fails → the ML/GenAI solution.

Where this sits on the ladder
This is the ladder applied. Every case starts by asking whether a cheaper rung works, then climbs only as far as needed — and the best solutions often combine rungs (a cheap classifier routing to an expensive LLM). Naming the combination, with its cost/latency tradeoffs, is what a customer is paying an FDE to do.

The scoping method

For any problem, walk the same four steps out loud — it’s exactly what an interviewer wants to hear:

  1. Frame it — what’s the actual input, output, and success metric? (classification? ranking? generation?)
  2. Try to break it with rules — why can’t deterministic code solve this? Name the reason (fuzzy meaning, unstructured input, too many patterns).
  3. Pick the rung(s) — the cheapest model(s) that fit the data type and the metric; often a combination.
  4. Wire it & guard it — the pipeline, the latency/cost budget, the failure modes, and how you’d evaluate it.

Case: “Search that understands meaning”

Problem: users search a help center but keyword search misses “I can’t sign in” when the article says “authentication failure.”
Why plain code fails: keyword/`LIKE` matching is lexical — it can’t see that two different phrasings mean the same thing.
Solution: embeddings + ANN vector search (rung 4 infra, no LLM needed). Embed articles and the query into the same space; return nearest neighbors by meaning. Add keyword (hybrid) search so exact terms and IDs still match.

Note the restraint
No LLM required — this is solved by embeddings + a vector index. Reaching for a chat model here would add cost and latency for nothing. (See Vector DBs.)

Case: “Answer questions from our documents”

Problem: employees ask natural-language questions that require synthesizing an answer from internal policy docs.
Why plain code (or search alone) fails: search returns documents; the user wants a synthesized answer with the right facts, phrased for their question.
Why not just an LLM: it doesn’t know your private docs and will hallucinate.
Solution: RAG — retrieve the relevant chunks (the search case above) and have an LLM answer from them, with citations. Combination of rung-4 infra (retrieval) + rung-4 generation. (See RAG.)

Case: “Route 10,000 support tickets a day” (a combination)

Problem: classify each ticket’s urgency and department, and draft a reply.
Why one tool doesn’t fit: classification is high-volume and cheap; reply-drafting needs generation. Using an LLM for everything is slow and expensive; using a classifier for everything can’t write.
Solution — combine rungs: a cheap classifier (fine-tuned SLM or even classical ML on ticket features) handles routing at volume; only tickets that need a reply go to an LLM + RAG for a grounded draft. Cheapest tool per sub-task.

The cheap model does the bulk work; the expensive model is reserved for the fraction that truly needs it. That routing pattern is the heart of cost-effective GenAI systems.

Case: “Catch fraudulent transactions”

Problem: flag fraud in real time from tabular transaction data.
Why plain rules fall short: hand-written rules (“flag > $5000 abroad”) are gamed and miss novel patterns — though they’re still a useful first filter.
Solution: tabular + labeled history → gradient-boosted trees (rung 2, CPU, fast, interpretable) for known fraud patterns, often combined with anomaly detection for novel ones. Definitely not an LLM — it’s numeric, latency-critical, and needs explainability for disputes. Threshold tuned by the cost of a false decline vs a missed fraud (see Metrics).

Case: “Pull structured data from messy documents”

Problem: turn invoices/contracts (PDFs, varied layouts) into structured fields.
Why plain code fails: regex/templates shatter on layout variety and wording.
Solution: an LLM with a strict output schema (rung 4) — feed the text, get validated JSON (see Prompt Engineering). If volume is huge and the layout is consistent, a cheaper specialized model (or a fine-tuned SLM) beats calling a frontier LLM per document. Validate every field before trusting it.

System-design talking points (for the whiteboard)

When asked to “design an AI system for X,” hit these dimensions — they show you think past the model:

  • Data flow — ingestion, offline indexing/training vs online serving (the split you know from data engineering).
  • The model choice — the rung(s), justified against cheaper options.
  • Latency & cost budget — p95 target, tokens/$, where caching and a cheap-model fast-path go.
  • Evaluation — the metric, the eval set, offline vs online (A/B).
  • Reliability — fallbacks, guardrails, monitoring/drift, rollback.
  • Human-in-the-loop — where a person reviews high-stakes outputs.

Interview Q&A

coreDesign a system to answer customer questions from our knowledge base.

This is RAG. Offline: chunk the KB, embed the chunks, and index them in a vector store with metadata (product, version, permissions). Online: embed the question, hybrid-retrieve the top chunks, re-rank, and have an LLM answer strictly from them with citations, saying “I don’t know” when unsupported. Around it: a task-specific eval set scoring faithfulness and relevance, guardrails and PII handling, caching for common questions, latency/cost budgets, and monitoring. I’d avoid fine-tuning facts in — RAG keeps answers fresh and citable.

trapWould you use an LLM to detect fraud in transactions?

No — that’s the wrong rung. Fraud detection is tabular, latency-critical, needs explainability for disputes, and has labeled history, which is exactly gradient-boosted-tree territory (often with anomaly detection for novel patterns). An LLM would be slower, costlier, non-deterministic, and weaker on numeric features. I might use an LLM only to enrich features from free-text fields, but the decision model stays a tree. Picking the cheap, interpretable rung here is the correct call.

seniorHow would you keep an LLM feature affordable at 10,000 requests/day?

Reserve the expensive model for the requests that need it. Put a cheap classifier or router first so simple/known cases are handled by rules, a small model, or a cache; only the hard fraction hits the big LLM. Add prompt/semantic caching for repeats, cap tokens, use a smaller or fine-tuned model where quality allows, and batch where possible. The pattern is a cost pyramid: cheap tools handle volume, the LLM handles the tail — combining rungs instead of using the costliest one everywhere.

coreA client wants “AI-powered search.” What do you actually build?

Usually semantic search with embeddings and a vector index — often no LLM at all. Embed documents and queries into the same space and return nearest neighbors by meaning, combined with keyword (hybrid) search so exact terms and IDs still match, plus a re-ranker for precision. If they actually want synthesized answers rather than a list of results, that’s a step up to RAG. I’d clarify which they need, because the cheaper embedding-only version solves a lot of “AI search” asks.

seniorWhen is combining models better than a single one?

When sub-tasks have different cost/quality profiles. A cheap classifier routing to an expensive generator (support triage), rules pre-filtering before a model (fraud), or a retriever feeding a generator (RAG) all let each component do what it’s best and cheapest at. The alternative — one big model doing everything — is simpler but usually slower, pricier, and less controllable. The art is decomposing the problem so the expensive rung only runs where it must.

Code: a hybrid triage router

The combination pattern in code: a cheap check handles the easy majority; the expensive LLM only runs on what’s left. This structure is most of cost-effective GenAI engineering.

pythoncheap-first routing: rules → classifier → LLM only when needed
# Combine rungs: reserve the expensive LLM for the fraction that truly needs it.
def handle_ticket(ticket: dict) -> dict:
    text = ticket["text"]

    # --- Rung 1: rules catch the obvious, for free. ---------------------------
    if any(k in text.lower() for k in ("refund", "cancel", "password reset")):
        return {"route": "self-service", "cost": 0}          # deflected, no model

    # --- Rung 2: a cheap classifier routes the bulk (CPU / small model). ------
    priority = classifier.predict([featurize(text)])[0]      # milliseconds, ~$0
    if priority in ("P2", "P3"):
        return {"route": f"queue-{priority}", "cost": 0}     # routed, still no LLM

    # --- Rung 4: ONLY urgent tickets get an LLM + RAG drafted reply. -----------
    context = retrieve(text)                                  # RAG: grounding docs
    draft = call_llm(system="Draft a support reply using ONLY the context.",
                     user=f"{text}\n\nContext:\n{context}")   # $$ per call
    return {"route": "agent-review", "draft": draft, "cost": "$$"}

# Most tickets never reach the LLM. You spent the least compute that solved each
# case — and can point to exactly why each tier exists. That's the FDE answer.
The pattern to remember
Great AI systems are rarely one model — they’re a pipeline that spends the least on each sub-task. Start from the problem, break it into pieces, and put the cheapest rung that works on each. That instinct is the whole job.
