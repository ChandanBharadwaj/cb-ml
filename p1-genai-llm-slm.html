GenAI · LLM · SLM

A transformer is the engine; a language model is what you get when you train one to predict the next token over the whole internet. This page is about what that unlocks — generation — and the knobs, limits, and cost trade-offs you’ll defend in every GenAI interview.

Where this sits on the ladder
Rung 4 — the top, and the most expensive. An LLM is astonishingly capable and the priciest tool in the box (GPU memory or $/token, higher latency). The whole art here is spending the least: pick the smallest model that clears the bar, and prefer prompting → RAG → fine-tuning in that order of cost.

Generative vs discriminative

A discriminative model learns a boundary: given input x, predict label y — “is this email spam?” A generative model learns the distribution of the data itself, so it can sample new data — “write an email.” LLMs are generative: they model P(next token | all previous tokens) and produce text one token at a time.

FDE framing
Discriminative = classify / score (rungs 2–3 territory, cheap). Generative = produce open-ended content (rung 4). If the task is really “pick one of N labels,” a classifier is cheaper and more reliable than asking an LLM to choose.

The autoregressive objective

Pretraining is deceptively simple: predict the next token, over and over, on trillions of tokens. That single objective forces the model to learn grammar, facts, reasoning patterns, and style — all as a side effect of getting good at “what comes next.” At inference, the model feeds its own output back in and repeats, which is why generation is sequential (and why it can’t be fully parallelized like the forward pass).

Generation is a loop: predict a token, append it, predict again. The model only ever answers one question — “what’s the next token?”

Decoding: how the next token is chosen

The model outputs a probability over the entire vocabulary. Decoding is the strategy for turning that distribution into an actual token — and it’s the knob you’ll tune most in production:

StrategyWhat it doesUse when
GreedyAlways take the highest-probability tokenDeterministic, factual, extraction tasks
Beam searchKeep the top-k running sequences, pick the best overallTranslation, where global fluency matters
TemperatureFlatten (high) or sharpen (low) the distribution before samplingDial creativity: low ≈ focused, high ≈ diverse
Top-kSample only from the k most likely tokensCut off the long tail of nonsense
Top-p (nucleus)Sample from the smallest set whose probability sums to pThe modern default for open-ended text
Low temperature concentrates probability on the top token (predictable); high temperature spreads it out (creative, but riskier).
Interview-ready knob settings
Extraction / classification / SQL / JSON: temperature 0 (greedy) — you want the same answer every time. Brainstorming / copywriting: temperature 0.7–1.0 with top-p ~0.9. Reaching for high temperature to “fix” wrong answers is a red flag — that’s a grounding problem (RAG), not a sampling problem.

Context window & the KV cache

The context window is how many tokens the model can attend to at once (4k → 200k+ depending on the model). Everything — system prompt, chat history, retrieved documents, and the answer being generated — must fit inside it. Because attention is O(n²), long contexts cost more compute and money, and quality can degrade in the middle (“lost in the middle”). To avoid recomputing attention for every prior token on each new step, models cache past keys/values — the KV cache (covered on the Inference Optimization page).

FDE takeaway
A bigger context window is not free and not always better. Stuffing 100k tokens of documents in is slower, pricier, and often worse than retrieving the right 2k tokens with RAG. Context is a budget, not a dumping ground.

LLM vs SLM — the cost decision

A Small Language Model (1–8B params — Phi, Gemma, Llama-8B, Mistral-7B) is the rung-4 equivalent of “do I really need the big one?” For a well-scoped task — classification, extraction, routing, a narrow chatbot — a fine-tuned SLM can match a frontier LLM at a fraction of the cost and latency, and it can run on-prem or even on-device (huge for regulated customers).

SLM (1–8B)LLM (70B–frontier)
Cost / latencyLow — cheap, fastHigh — $/token, slower
HostingOn-prem / on-device / single GPUMulti-GPU or hosted API
Best atNarrow, well-defined tasks (esp. fine-tuned)Broad reasoning, open-ended, zero-shot generality
Data privacyEasy to keep in your VPCOften sends data to a vendor
FDE fitShip cheap, private, per-customerPrototype fast, hardest tasks
Ladder rule (inside rung 4)
Even at the top rung there’s a ladder: try a hosted frontier LLM to prove the task is solvable, then shrink to the smallest SLM that still passes your evals for production. Prototyping cost ≠ production cost.

Why LLMs hallucinate

An LLM is trained to produce plausible next tokens, not true ones. It has no built-in notion of “I don’t know” — so when the answer isn’t in its weights, it confidently makes something up that sounds right. This is a feature of the objective, not a bug you can fully prompt away. The production fixes:

  • Ground it with retrieved facts (RAG) so the answer comes from your data, not its memory.
  • Constrain it — structured output, allowed-value lists, “say ‘unknown’ if unsure.”
  • Verify it — cite sources, check answers against the context, use an eval/judge.

Interview Q&A

coreWhat does “autoregressive” mean for an LLM?

It generates one token at a time, and each new token is conditioned on all the tokens before it — including the ones it just produced. Training maximizes P(next token | previous tokens) over huge corpora; inference feeds the model’s own output back in and repeats. That sequential dependency is why generation latency scales with output length and can’t be parallelized the way the forward pass can.

coreTemperature vs top-p — how do you set them?

Temperature rescales the distribution (low = sharp/deterministic, high = flat/creative); top-p (nucleus) samples from the smallest set of tokens covering probability p, clipping the unlikely tail. In practice: for factual/extraction work use temperature 0. For creative work use temperature ~0.7–1.0 with top-p ~0.9. I generally tune one, not both, to keep behavior predictable.

seniorWhen would you choose an SLM over a frontier LLM?

When the task is narrow and high-volume (classification, extraction, routing, a scoped assistant), when latency or cost per call matters, or when data can’t leave the customer’s environment. A fine-tuned 7B model can match a frontier model on a specific task for a fraction of the cost and run on a single GPU or on-device. I’d prototype on a hosted LLM to prove feasibility, then distill/fine-tune down to the smallest model that passes evals.

trapThe model keeps inventing facts. Do you raise or lower temperature?

Neither — temperature is the wrong lever. Hallucination is a grounding problem: the fact isn’t reliably in the weights, so the model fills the gap. Lowering temperature just makes it confidently wrong more consistently. The fix is to supply the facts (RAG), constrain the output, and let it say “I don’t know.” Temperature only controls randomness, not truth.

seniorWhy can’t you speed up generation just by adding GPUs like the forward pass?

Because generation is inherently sequential — token t+1 depends on token t, which the model just produced. You can parallelize within a step (and across many requests via batching), but you can’t compute a sequence’s tokens simultaneously. That’s why output length drives latency, and why tricks like speculative decoding and KV-caching exist to shave per-token cost rather than parallelize the sequence.

coreWhat lives in the context window, and why does size matter?

Everything the model sees on this call: system prompt, conversation history, any retrieved documents, and the tokens generated so far. Since attention is quadratic in length, a larger window costs more compute, more money, and more latency, and models can lose track of material buried in the middle. The skill is fitting the right context, not the most — which is exactly why retrieval beats “paste everything.”

Code: controlling generation with Hugging Face

The same prompt, two behaviors — deterministic vs creative — driven entirely by decoding parameters. This is the exact dial you tune when moving a feature from “demo” to “reliable.”

pythondecoding parameters with model.generate()
# Controlling how an LLM decodes text. The MODEL is fixed; only the KNOBS change.
# pip install transformers torch
from transformers import AutoModelForCausalLM, AutoTokenizer
import torch

name = "Qwen/Qwen2.5-0.5B-Instruct"          # a tiny SLM — runs on CPU for demos
tok   = AutoTokenizer.from_pretrained(name)
model = AutoModelForCausalLM.from_pretrained(name)

prompt = "In one sentence, explain what a vector database is."
inputs = tok(prompt, return_tensors="pt")

# --- Mode A: DETERMINISTIC (facts, extraction, SQL, JSON) ---------------------
# do_sample=False -> greedy: always pick the top token. Same input => same output.
factual = model.generate(
    **inputs,
    max_new_tokens=60,
    do_sample=False,          # greedy decoding — no randomness
)
print("FACTUAL:", tok.decode(factual[0], skip_special_tokens=True))

# --- Mode B: CREATIVE (brainstorming, copy) ----------------------------------
# Sample from a flattened distribution, clipped to the nucleus (top-p).
creative = model.generate(
    **inputs,
    max_new_tokens=60,
    do_sample=True,           # turn on sampling
    temperature=0.9,          # flatten the distribution -> more diverse
    top_p=0.9,                # nucleus: only sample from tokens summing to 90% prob
    top_k=50,                 # also cap to the 50 most likely tokens
    repetition_penalty=1.1,   # gently discourage repeating itself
)
print("CREATIVE:", tok.decode(creative[0], skip_special_tokens=True))

# Rule of thumb: start at temperature 0 and only add randomness if the task
# genuinely benefits from variety. Determinism is a feature in production.
Connect it to what you know
Think of decoding params like log levels or feature flags: same binary, different runtime behavior. You wouldn’t ship a service with DEBUG logging on by accident — likewise, don’t ship a data-extraction endpoint with temperature=1.0. Pin it to 0 and make it boring.
