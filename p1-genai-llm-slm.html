GenAI, LLMs & SLMs

A transformer is the engine; a language model is what you get when you train one to predict the next token over the whole internet. This page is about what that unlocks — generation — and the knobs, limits, scale, and cost trade-offs you’ll defend in every Generative AI interview.

Generative vs discriminative

What it is

A discriminative model learns a boundary between classes — “is this email spam?” A generative model learns the distribution of the data itself, so it can sample new data — “write an email.” Large Language Models are generative.

A discriminative model, given input x, predicts label y. A generative model instead models how the data is produced, which lets it create fresh examples. LLMs are generative in a very specific way: they model P(next token | all previous tokens) and produce text one token at a time. That’s the whole trick — turning “understand language” into the repeatable task of “predict what comes next.”

Bottom line
Discriminative = classify / score (rungs 2–3, cheap). Generative = produce open-ended content (rung 4). If the task is really “pick one of N labels,” a classifier is cheaper and more reliable than asking an LLM to choose.

The autoregressive objective

What it is

Autoregressive means the model generates one token at a time, and each new token is conditioned on everything before it — including the tokens it just produced. Both training and generation are built on this single objective: predict the next token.

Pretraining is deceptively simple: predict the next token, over and over, on trillions of tokens. That one objective forces the model to learn grammar, facts, reasoning patterns, and style — all as a side effect of getting good at “what comes next.” At inference, the model feeds its own output back in and repeats, which is why generation is sequential (and why it can’t be fully parallelized like the forward pass on the neural-networks page).

Generation is a loop: predict a token, append it, predict again. The model only ever answers one question — “what’s the next token?”
Bottom line
One objective — next-token prediction — does everything: it’s how the model learns and how it generates. Its sequential nature is also why output length drives latency.

Decoding: how the next token is chosen

What it is

Decoding is the strategy that turns the model’s probability distribution over the vocabulary into one actual token. The model gives you the odds; decoding decides how you pick — and it’s the knob you tune most in production.

StrategyWhat it doesUse when
GreedyAlways take the highest-probability tokenDeterministic, factual, extraction tasks
Beam searchKeep the top-k running sequences, pick the best overallTranslation, where global fluency matters
TemperatureFlatten (high) or sharpen (low) the distribution before samplingDial creativity: low ≈ focused, high ≈ diverse
Top-kSample only from the k most likely tokensCut off the long tail of nonsense
Top-p (nucleus)Sample from the smallest set whose probability sums to pThe modern default for open-ended text
Low temperature concentrates probability on the top token (predictable); high temperature spreads it out (creative, but riskier).
Interview-ready knob settings
Extraction / classification / SQL / JSON: temperature 0 (greedy) — you want the same answer every time. Brainstorming / copywriting: temperature 0.7–1.0 with top-p ~0.9. Reaching for high temperature to “fix” wrong answers is a red flag — that’s a grounding problem (RAG), not a sampling problem.
Bottom line
The model’s probabilities are fixed; decoding chooses how boldly you sample from them. Determinism (temperature 0) for facts, controlled randomness (top-p) for creativity.

Context window & the KV cache

What it is

The context window is the maximum number of tokens the model can attend to at once. The KV cache (Key–Value cache) is an optimization that stores the attention keys and values of past tokens so each new token is cheap to generate.

Everything the model uses on a call — system prompt, chat history, retrieved documents, and the answer being generated — must fit inside the context window (4k → 200k+ tokens depending on the model). Because attention is O(n²), long contexts cost more compute and money, and quality can degrade in the middle (the “lost in the middle” effect). To avoid recomputing attention over every prior token at each step, models cache past keys and values — the KV cache — which is covered in depth in Part 3 → Inference Optimization.

Bottom line
A bigger context window is not free and not always better. Retrieving the right 2k tokens usually beats stuffing in 100k. Context is a budget, not a dumping ground.

Scaling laws & model size

What it is

Scaling laws are the empirical finding that a model’s loss falls predictably as you grow three things together — parameters, training data, and compute. “Model size” is the parameter count: a “7B” model has seven billion learned weights.

This is why language models got dramatically better: not one clever trick, but making the same transformer bigger, trained on more tokens, with more compute — and the improvement follows a smooth curve you can extrapolate. Two consequences you should be able to name. Emergence: some abilities (multi-step reasoning, in-context learning) appear only past a certain scale, seemingly “switching on.” The compute-optimal balance (the Chinchilla finding): for a fixed compute budget, you shouldn’t just grow parameters — you must grow training data proportionally, which is why smaller-but-well-trained models can beat larger under-trained ones.

Loss drops smoothly with scale, but with diminishing returns — doubling capability costs far more than double. That curve is exactly why the size decision below matters.
Bottom line
Scale buys capability predictably but with steep diminishing returns. The engineering question is never “biggest?” — it’s “what’s the smallest model on this curve that clears my bar?”

LLM vs SLM — the cost decision

What it is

A Small Language Model (SLM) is a compact model (~1–8 billion parameters — Phi, Gemma, Llama-8B, Mistral-7B) chosen when you don’t need a frontier Large Language Model (LLM). It’s the rung-4 version of “do I really need the big one?”

For a well-scoped task — classification, extraction, routing, a narrow chatbot — a fine-tuned SLM can match a frontier LLM at a fraction of the cost and latency, and it can run on-premises or even on-device (huge for regulated customers). The scaling curve above is what makes this real: past the point of diminishing returns, the extra parameters of a frontier model add little on a narrow task but cost a lot to serve.

Ladder rule (inside rung 4)
Even at the top rung there’s a ladder: try a hosted frontier LLM to prove the task is solvable, then shrink to the smallest SLM that still passes your evals for production. Prototyping cost ≠ production cost.

Why LLMs hallucinate

What it is

A hallucination is a confident, fluent answer that is simply false. It happens because an LLM is trained to produce plausible next tokens, not true ones — it has no built-in notion of “I don’t know.”

When the answer isn’t reliably in its weights, the model fills the gap with something that sounds right. This is a property of the next-token objective, not a bug you can fully prompt away. The production fixes:

  • Ground it with retrieved facts (Retrieval-Augmented Generation) so the answer comes from your data, not its memory.
  • Constrain it — structured output, allowed-value lists, “say ‘unknown’ if unsure.”
  • Verify it — cite sources, check answers against the context, use an eval / judge.
Bottom line
Hallucination is a grounding problem, not a randomness problem. The cure is retrieval, constraints, and verification — which is exactly what Part 2 is about.

Interview Q&A

coreWhat does “autoregressive” mean for an LLM?

It generates one token at a time, and each new token is conditioned on all the tokens before it — including the ones it just produced. Training maximizes P(next token | previous tokens) over huge corpora; inference feeds the model’s own output back in and repeats. That sequential dependency is why generation latency scales with output length and can’t be parallelized the way the forward pass can.

coreTemperature vs top-p — how do you set them?

Temperature rescales the distribution (low = sharp/deterministic, high = flat/creative); top-p (nucleus) samples from the smallest set of tokens covering probability p, clipping the unlikely tail. In practice: for factual/extraction work use temperature 0. For creative work use temperature ~0.7–1.0 with top-p ~0.9. I generally tune one, not both, to keep behavior predictable.

seniorWhat are scaling laws, and why do they matter for a build decision?

Scaling laws say loss improves predictably as parameters, data, and compute grow together — which is why bigger models got better and why some abilities only emerge past a certain scale. They matter because the curve has diminishing returns: on a narrow task, a frontier model’s extra parameters buy little but cost a lot to serve. The compute-optimal (Chinchilla) result adds that you must scale data with parameters, so a smaller, well-trained model often beats a bigger, under-trained one — the basis for choosing an SLM.

seniorWhen would you choose an SLM over a frontier LLM?

When the task is narrow and high-volume (classification, extraction, routing, a scoped assistant), when latency or cost per call matters, or when data can’t leave the customer’s environment. A fine-tuned 7B model can match a frontier model on a specific task for a fraction of the cost and run on a single GPU or on-device. I’d prototype on a hosted LLM to prove feasibility, then distill/fine-tune down to the smallest model that passes evals.

trapThe model keeps inventing facts. Do you raise or lower temperature?

Neither — temperature is the wrong lever. Hallucination is a grounding problem: the fact isn’t reliably in the weights, so the model fills the gap. Lowering temperature just makes it confidently wrong more consistently. The fix is to supply the facts (RAG), constrain the output, and let it say “I don’t know.” Temperature only controls randomness, not truth.

coreWhat lives in the context window, and why does size matter?

Everything the model sees on this call: system prompt, conversation history, any retrieved documents, and the tokens generated so far. Since attention is quadratic in length, a larger window costs more compute, more money, and more latency, and models can lose track of material buried in the middle. The skill is fitting the right context, not the most — which is exactly why retrieval beats “paste everything.”

Code: controlling generation with Hugging Face

The same prompt, two behaviors — deterministic vs creative — driven entirely by decoding parameters. This is the exact dial you tune when moving a feature from “demo” to “reliable.”

pythondecoding parameters with model.generate()
# Controlling how an LLM decodes text. The MODEL is fixed; only the KNOBS change.
# pip install transformers torch
from transformers import AutoModelForCausalLM, AutoTokenizer
import torch

name = "Qwen/Qwen2.5-0.5B-Instruct"          # a tiny SLM — runs on CPU for demos
tok   = AutoTokenizer.from_pretrained(name)
model = AutoModelForCausalLM.from_pretrained(name)

prompt = "In one sentence, explain what a vector database is."
inputs = tok(prompt, return_tensors="pt")

# --- Mode A: DETERMINISTIC (facts, extraction, SQL, JSON) ---------------------
# do_sample=False -> greedy: always pick the top token. Same input => same output.
factual = model.generate(
    **inputs,
    max_new_tokens=60,
    do_sample=False,          # greedy decoding — no randomness
)
print("FACTUAL:", tok.decode(factual[0], skip_special_tokens=True))

# --- Mode B: CREATIVE (brainstorming, copy) ----------------------------------
# Sample from a flattened distribution, clipped to the nucleus (top-p).
creative = model.generate(
    **inputs,
    max_new_tokens=60,
    do_sample=True,           # turn on sampling
    temperature=0.9,          # flatten the distribution -> more diverse
    top_p=0.9,                # nucleus: only sample from tokens summing to 90% prob
    top_k=50,                 # also cap to the 50 most likely tokens
    repetition_penalty=1.1,   # gently discourage repeating itself
)
print("CREATIVE:", tok.decode(creative[0], skip_special_tokens=True))

# Rule of thumb: start at temperature 0 and only add randomness if the task
# genuinely benefits from variety. Determinism is a feature in production.
