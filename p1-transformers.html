What a transformer is

What it is

A transformer is a neural network built around self-attention — a mechanism that lets every token in a sequence look directly at every other token, in parallel. Introduced in the 2017 paper “Attention Is All You Need,” it replaced the step-by-step recurrence of earlier sequence models and became the backbone of essentially every modern language model.

Stripped down, a transformer is a stack of identical blocks, each combining self-attention (tokens mixing information) with a small per-token neural network. Its inputs are the token embeddings and positional encodings from the previous page; its output is a context-aware vector for every token. Everything else — chat models, code models, embeddings models — is a configuration of this one idea.

Bottom line
A transformer = embeddings + positional info, run through a stack of self-attention blocks. Master attention and you understand the whole family.

Why transformers replaced RNNs

What it is

Before transformers, sequences were handled by Recurrent Neural Networks (RNNs) and their Long Short-Term Memory (LSTM) variants, which read a sequence one token at a time while carrying a running hidden state. Transformers threw out that recurrence.

RNNs had two crippling problems. (1) No parallelism: token t needs token t-1’s hidden state, so training can’t be spread across the sequence — it’s slow. (2) Fading memory: long-range dependencies get squeezed through many steps and the signal vanishes. Self-attention fixes both at once: every token attends to every other token directly, in parallel, with a constant-length path between any two positions. That single change unlocked GPU-parallel training on internet-scale data — which is what made today’s LLMs possible.

One-line answer
“Transformers replace sequential recurrence with parallel self-attention, so every token can attend to every other token in O(1) path length — enabling long-range context and massively parallel training.”

Self-attention, derived

What it is

Self-attention rebuilds each token’s representation as a weighted average of every token’s information, where the weights measure how relevant each other token is. It’s how a transformer gives a word meaning in context.

For each token we compute three vectors by multiplying its embedding by learned matrices:

  • Query (Q) — “what am I looking for?”
  • Key (K) — “what do I offer?”
  • Value (V) — “what I’ll actually pass on if attended to.”

A token’s new representation is a weighted average of every token’s Value, where the weight is how well this token’s Query matches that token’s Key. The full formula:

Scaled dot-product attention
Attention(Q, K, V) = softmax( Q·Kᵀ / √dk ) · V
Every token builds its meaning from a weighted blend of the others. Here “sat” leans on its subject “cat” and object “mat.”

Why divide by √dk?

A classic interview follow-up. The dot product Q·Kᵀ sums over dk dimensions, so its variance grows with dk. Large values push softmax into saturated regions where gradients vanish and it behaves like a hard argmax. Dividing by √dk keeps the scores unit-variance, so the softmax stays smooth and trainable.

Show the causal-mask detail (decoders)

In a decoder (GPT-style, generating left-to-right) a token must not see the future. Before the softmax we add a causal mask that sets all “future” scores to -∞, so their softmax weight becomes 0. That’s the only difference that makes attention autoregressive.

Bottom line
Self-attention = match every Query to every Key, softmax the scores, and mix the Values. It’s all matrix multiplication, which is why it runs fast on a GPU and scales to huge models.

Multi-head attention

What it is

Multi-head attention runs several self-attention operations in parallel, each with its own Q/K/V projections, then concatenates their outputs. Each “head” can focus on a different kind of relationship.

One attention pattern isn’t enough — a token may need syntactic and semantic relationships at once. So we run attention h times in parallel (e.g. 8–96 heads), each with its own small Q/K/V projections, then concatenate and project back. Each head can specialise: one tracks subject→verb agreement, another tracks which pronoun refers to which noun, and so on. The total compute is about the same as one big attention, but the representation is far richer.

Split into h heads → each attends independently → concatenate → linear projection. Same total compute, far richer representations.
Bottom line
One head sees one kind of relationship; many heads see many at once. Concatenate and project, and the model captures syntax, meaning, and reference simultaneously.

The transformer block

What it is

A transformer block is the repeating unit of the architecture: multi-head self-attention followed by a small per-token Feed-Forward Network (FFN), each wrapped in a residual connection and layer normalization. Stack N of them and you have the model.

The block has two sub-layers. First, multi-head self-attention lets tokens exchange information. Second, a position-wise feed-forward network — a tiny MLP applied to each token independently — transforms each token’s representation. Each sub-layer is wrapped in a residual connection (add the input back to the output) and layer normalization, the same tricks from the neural-networks page that let you stack dozens of layers without the gradient degrading. GPT-3, for example, stacks 96 of these blocks.

Residual connections (green) let gradients flow straight through, so you can stack dozens of blocks without the signal degrading.
Bottom line
Attention mixes tokens; the feed-forward network processes each one; residuals and normalization keep a deep stack trainable. Repeat the block N times and you have a transformer.

Encoder vs decoder vs encoder-decoder

What it is

Transformers come in three flavours depending on what each token is allowed to attend to: encoder-only (see everything, for understanding), decoder-only (see only the past, for generation), and encoder-decoder (understand one sequence, generate another).

TypeAttentionGood atExamples
Encoder-onlyBidirectional (sees full context)Understanding: classification, embeddings, retrievalBERT, sentence-transformers
Decoder-onlyCausal (left-to-right)Generation: chat, completion, codeGPT, LLaMA, Mistral
Encoder-decoderEncoder bidirectional + decoder cross-attendsSeq-to-seq: translation, summarizationT5, BART
Rule of thumb
Need a vector that represents text (search, RAG, classification)? Reach for an encoder. Need to produce text? Reach for a decoder. Almost every chat LLM today is decoder-only — which is exactly where the next page picks up.

Interview Q&A

coreWalk me through self-attention.

Project each token embedding into Query, Key and Value vectors. Score every token pair with the dot product of one token’s Query and another’s Key, scale by 1/√dk, softmax the scores into weights, then take the weighted sum of Values. That weighted sum is the token’s new, context-aware representation. It’s done for all tokens at once as matrix multiplications, which is why it’s GPU-friendly.

trapWhy the √dk scaling — what breaks without it?

Without scaling, dot products grow with dimension (variance ∝ dk). Large logits saturate the softmax → near-zero gradients → the model can’t learn, and attention collapses to a hard pick of one token. Dividing by √dk normalizes the score variance to ~1, keeping softmax in its sensitive, differentiable range.

coreWhat’s the computational complexity of self-attention, and why does it matter?

O(n²·d) in sequence length n — every token attends to every other, so the attention matrix is n×n. That quadratic cost is exactly why long context is expensive and why techniques like FlashAttention, sliding-window attention, and KV-caching exist (see Part 3 → Inference Optimization).

coreWhy did transformers beat RNNs?

RNNs process tokens sequentially, so they can’t parallelize across a sequence and they lose long-range signal through many steps. Self-attention connects every token to every other directly and in parallel, giving constant-length paths for long-range dependencies and enabling GPU-parallel training on massive corpora — the combination that made large language models feasible.

seniorEncoder-only vs decoder-only — which would you pick for semantic search, and why?

Encoder-only (e.g. a sentence-transformer). Search needs a single, high-quality embedding per document, and bidirectional attention lets every token see the full context, producing better representations. Decoder-only models are optimized to generate the next token, not to summarize meaning into one vector — though you can pool their hidden states, encoders are usually cheaper and stronger for pure retrieval.

trapWhat does multi-head attention buy you over a single larger head?

Multiple heads let the model attend to different relationships in different representation subspaces at the same time — one head for syntax, another for coreference, another for position — then combine them. A single head of the same total size can only express one weighted-average pattern per token, so it can’t separate those signals as cleanly.

Code: attention from scratch

The mechanics in NumPy — no framework magic, so you can see every step. This is the exact thing an interviewer may ask you to whiteboard.

pythonscaled dot-product self-attention (NumPy)
# Scaled dot-product self-attention, implemented from scratch to show the mechanics.
# No deep-learning framework — just NumPy — so every step is visible.
import numpy as np

def softmax(x, axis=-1):
    # Subtract the max for numerical stability (avoids overflow in exp).
    x = x - x.max(axis=axis, keepdims=True)
    e = np.exp(x)
    return e / e.sum(axis=axis, keepdims=True)

def self_attention(X, W_q, W_k, W_v, causal=False):
    """
    X   : (seq_len, d_model)  input token embeddings (one row per token)
    W_* : (d_model, d_k)      learned projection matrices for Q, K, V
    Returns: (seq_len, d_k)   context-aware representation for each token
    """
    # 1) Project each token into Query, Key, Value spaces.
    Q = X @ W_q                      # (seq_len, d_k) — "what each token is looking for"
    K = X @ W_k                      # (seq_len, d_k) — "what each token offers"
    V = X @ W_v                      # (seq_len, d_k) — "what each token passes on"

    d_k = Q.shape[-1]

    # 2) Score every query against every key: how well do they match?
    scores = Q @ K.T                 # (seq_len, seq_len)

    # 3) Scale by sqrt(d_k) so the softmax doesn't saturate (the key trick).
    scores = scores / np.sqrt(d_k)

    # 4) Optional causal mask: a token must not attend to future tokens (decoders).
    if causal:
        seq_len = scores.shape[0]
        # Upper-triangular positions (above the diagonal) are "the future".
        mask = np.triu(np.ones((seq_len, seq_len)), k=1).astype(bool)
        scores[mask] = -np.inf       # -inf -> softmax weight becomes exactly 0

    # 5) Turn scores into probabilities (each row sums to 1).
    weights = softmax(scores, axis=-1)   # (seq_len, seq_len)

    # 6) Weighted sum of Values = the new representation of each token.
    return weights @ V, weights          # output (seq_len, d_k), plus weights to inspect

# --- tiny demo ---------------------------------------------------------------
np.random.seed(0)
X = np.random.randn(4, 8)            # 4 tokens, 8-dim embeddings
Wq, Wk, Wv = (np.random.randn(8, 8) for _ in range(3))
out, attn = self_attention(X, Wq, Wk, Wv, causal=True)
print("output shape:", out.shape)   # (4, 8)
print("attention row 0 (sums to 1):", attn[0].round(3))
