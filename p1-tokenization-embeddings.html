Tokenization, Embeddings & NLP

A neural network only does math on numbers — so the first job of any language model is turning text into numbers, and back. This page covers that pipeline end to end: what Natural Language Processing is, how text is split into tokens, how tokens become meaning-carrying embeddings, and how models are told the order of the words.

What Natural Language Processing is

What it is

Natural Language Processing (NLP) is the field of getting computers to read, understand, and produce human language. Everything from spam filters to search to chatbots is NLP; modern NLP does it by converting text into numbers, doing math, and converting back.

The hard part is that language is unstructured — no tidy columns, huge vocabulary, meaning that depends on order and context. For decades NLP relied on hand-built rules and features; today it is dominated by neural networks that learn from raw text. But every one of those models starts with the same two steps this page is about: tokenize the text into units, then embed those units as vectors. Get those right and the transformer on the next page does the rest.

Bottom line
NLP is “machine learning for language.” The pipeline is always the same: text → tokens → embeddings → model → tokens → text.

Tokenization

What it is

Tokenization is the step that splits text into tokens and maps each to an integer ID. A token is usually a subword — a common word stays whole, a rare word breaks into reusable pieces — so a small fixed vocabulary can represent any string.

Models don’t see letters; they see integer token IDs. Modern language models use subword tokenization — Byte-Pair Encoding (BPE), WordPiece, or SentencePiece — which keeps the vocabulary small (~30k–100k) while still covering typos and words it never saw in training. BPE works by starting from characters and repeatedly merging the most frequent adjacent pair into a new token, so high-frequency chunks become single tokens and rare words fall back to smaller pieces.

“tokenization” is rare, so it splits into token+iza+tion; common words stay whole. Each token maps to an integer ID.
FDE gotcha: tokens ≠ words
English averages ~1.3 tokens per word; code and non-Latin scripts cost more. Since you pay per token and context windows are measured in tokens, a “4k-word” document can blow a budget. Always estimate tokens, not words.
Bottom line
Tokenization is the bridge from text to integers. Subword schemes like BPE give a small vocabulary that still represents anything — and the token, not the word, is the unit you count and pay for.

Embeddings

What it is

An embedding is a learned vector that represents a token’s meaning as a point in space, arranged so that tokens with similar meaning sit close together. A token ID indexes into an embedding table to fetch its vector.

Once text is a list of token IDs, each ID looks up a row in a big learned matrix — the embedding table — giving a vector of, say, 768–4096 numbers. During training these vectors organise themselves into a semantic space: “king” lands near “queen,” “Paris” near “France.” This is the same idea as a feature vector from earlier pages, except the features are learned rather than hand-designed. Those vectors are also what you store in a vector database to power semantic search and Retrieval- Augmented Generation.

Bottom line
An embedding turns a discrete token into a continuous vector of meaning. Similar meanings → nearby vectors — the property every downstream language task relies on.

Positional encoding

What it is

Positional encoding is the extra signal that tells the model the order of the tokens. Attention (next page) treats its input as an unordered set, so without this a model couldn’t tell “dog bites man” from “man bites dog.”

The embedding says what a token is; positional encoding says where it sits. The original transformer added fixed sinusoidal signals — a bank of sine and cosine waves at different frequencies, giving every position a unique fingerprint. Most models since 2023 (LLaMA, most open models) instead use Rotary Position Embedding (RoPE), which rotates the query and key vectors by an angle proportional to position. The elegant result: attention ends up depending on the relative distance between tokens, and it extrapolates better to sequences longer than those seen in training.

Position is encoded as a bank of sinusoids at different frequencies. Every position gets a unique fingerprint; nearby positions get similar ones.
Bottom line
Embeddings encode meaning; positional encoding encodes order. Modern models prefer RoPE because relative-position awareness generalises to longer contexts — which is why it underpins today’s long-context LLMs.

Interview Q&A

coreWhy subword tokenization instead of splitting on words?

Word-level vocabularies explode in size and still fail on unseen words, typos, and other languages. Subword schemes (BPE, WordPiece, SentencePiece) keep a small fixed vocabulary while representing any string: common words are single tokens, rare ones decompose into reusable pieces. That balances vocabulary size against sequence length.

trapA user pastes a 4,000-word document. Why might it not fit a 4k-token context window?

Because tokens aren’t words. English runs ~1.3 tokens per word, and code, punctuation, or non-Latin scripts push it higher — so 4,000 words can be 5,000–6,000+ tokens. Context limits and billing are in tokens, so you always estimate in tokens, not words.

coreWhat is an embedding, in one line?

A learned vector that places a token (or a whole sentence) in a semantic space where similar meanings are close together. It’s how discrete text becomes continuous numbers a network can do math on — and what you index in a vector database for semantic search.

trapIf a model processes tokens in parallel, how does it know word order?

It doesn’t, until you add positional information. On its own, attention treats the input as an unordered set. Positional encodings — sinusoidal, learned, or RoPE — inject order so the model can distinguish “dog bites man” from “man bites dog.”

seniorWhy do modern LLMs use RoPE instead of the original sinusoidal encodings?

Sinusoidal/learned absolute encodings are added to the input once. RoPE instead rotates the query and key vectors by an angle tied to position, so the attention score between two tokens depends on their relative offset. That generalises better to positions unseen in training and makes context-length extension (e.g. NTK / YaRN scaling) tractable — which is why almost all 2023+ open models adopted it.

Code: tokens & embeddings with Hugging Face

Text in, vectors out — the real-world version of this whole pipeline. Here we tokenize two sentences and pull a sentence embedding from a small pretrained model, then check that two paraphrases land close together.

pythontokenizer + embeddings with Hugging Face
# Turn text into token IDs, then into a meaning vector, using a pretrained model.
# pip install transformers torch
from transformers import AutoTokenizer, AutoModel
import torch

model_name = "sentence-transformers/all-MiniLM-L6-v2"   # a small encoder — CPU-friendly
tokenizer = AutoTokenizer.from_pretrained(model_name)
model = AutoModel.from_pretrained(model_name)

sentences = ["The cat sat on the mat.", "A feline rested on a rug."]

# 1) Tokenize: text -> token IDs (+ attention mask for padding).
enc = tokenizer(sentences, padding=True, truncation=True, return_tensors="pt")
print("token IDs:\n", enc["input_ids"])

# 2) Forward pass through the model (no gradients needed for inference).
with torch.no_grad():
    out = model(**enc)                       # last_hidden_state: (batch, seq_len, hidden)

# 3) Mean-pool token vectors into one sentence embedding (mask out padding).
mask = enc["attention_mask"].unsqueeze(-1)   # (batch, seq_len, 1)
summed = (out.last_hidden_state * mask).sum(1)
counts = mask.sum(1).clamp(min=1e-9)
embeddings = summed / counts                 # (batch, hidden) — one vector per sentence

# 4) Cosine similarity: the two paraphrases should score high despite different words.
a, b = torch.nn.functional.normalize(embeddings, dim=1)
print("cosine similarity:", float(a @ b))    # ~0.7+ — semantic match, not word overlap
