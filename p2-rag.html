RAG — Retrieval-Augmented Generation

The most-tested GenAI topic in modern interviews, and the workhorse of real deployments. RAG gives an LLM an open book: instead of answering from memory (and hallucinating), it retrieves your relevant documents at query time and answers from them. This is how you put a model on top of private, fresh, or proprietary data without training it.

Where this sits on the ladder
The middle LLM lever: prompt → RAG → fine-tune. Reach for RAG when the problem is “the model doesn’t know our stuff” — internal docs, changing data, facts that must be cited. It adds knowledge without changing weights, so it’s far cheaper and more maintainable than fine-tuning, and updates instantly when your data changes.

Why RAG instead of fine-tuning

A common trap is “our LLM doesn’t know our docs, so let’s fine-tune it.” Usually wrong. Fine-tuning teaches behavior and style; it’s a poor way to inject facts — they’re expensive to bake in, go stale, can’t be cited, and are hard to update. RAG injects facts at query time: change a document, and the next answer reflects it.

RAGFine-tuning
Best forFacts, knowledge, fresh/private dataBehavior, format, tone, a skill
Update costInstant — re-index a docRe-train the model
CitationsYes — you know the sourceNo — knowledge is opaque in weights
Setup costLow (no GPU training)High (data + training)
The line to say
“RAG for knowledge, fine-tuning for behavior — and they compose. If the model needs to know our data, that’s retrieval; if it needs to act or sound a certain way, that’s fine-tuning.”

The pipeline, end to end

RAG is two phases: an offline indexing phase (done once / on updates) and an online query phase (per request).

Index once: split docs → embed → store. Per query: embed the question → retrieve nearest chunks → re-rank → hand the best ones + the question to the LLM → answer grounded in your data.

Chunking strategies

You don’t embed whole documents — you split them into chunks. Chunking is the most under-rated lever in RAG: get it wrong and retrieval fails no matter how good your model is.

  • Fixed-size (e.g. 300–500 tokens) with overlap (10–20%) so a sentence split across a boundary still appears whole in one chunk. Simple, strong default.
  • Semantic / recursive — split on structure (headings, paragraphs) so a chunk is a coherent idea, not an arbitrary window.
  • Too big → chunks contain lots of irrelevant text, diluting the match and wasting context. Too small → chunks lose the surrounding meaning. Tune to your docs.
FDE gotcha
Chunk size trades recall against precision and directly drives token cost. It’s the first knob to tune when RAG “doesn’t work” — usually before touching the model or the prompt.

Hybrid search & re-ranking

Two retrieval signals, better together:

  • Dense (semantic) — embedding similarity. Finds meaning even when words differ (“car” ↔ “automobile”). Weak on exact terms, IDs, rare keywords.
  • Sparse (lexical, BM25) — keyword match. Nails exact terms, codes, names. Blind to synonyms.
  • Hybrid — combine both scores (e.g. reciprocal rank fusion). The production default, because real queries need both.

Then re-rank: retrieve a generous top-k (say 20) cheaply, and use a cross-encoder re-ranker to score each candidate against the query jointly and keep the best 3–5. Retrieval is recall-oriented and fast; re-ranking is precision-oriented and accurate. Together they beat either alone.

Don’t over-engineer
Basic hybrid-search RAG with good chunking and re-ranking solves the large majority of cases. Reach for agentic/graph variants only when you can point to a failure mode they fix — naming them shows breadth, but knowing when not to shows judgment.

Failure modes (name these in an interview)

  • Lost in the middle — LLMs attend best to the start and end of context; relevant chunks buried in the middle get ignored. Fix: retrieve fewer, better chunks; re-rank so the best sit first.
  • Retrieval miss — the answer isn’t in the retrieved chunks at all (bad chunking, wrong embedding, query mismatch). The model then hallucinates or says it can’t find it. This is the #1 cause of “RAG is wrong.”
  • Multi-hop — the answer requires combining facts from several documents; a single retrieval can’t assemble them. Fix: query decomposition / agentic retrieval.
  • Stale or conflicting sources — old and new docs disagree; the model may pick the wrong one. Fix: metadata (recency) filtering, deduplication, source priority.
  • Hallucination despite good context — the answer’s in the chunks but the model ignores it. Fix: instruct “answer only from the context; if absent, say so,” and cite.

Evaluating RAG

You can’t improve what you don’t measure, and RAG has two things to measure separately:

  • Retrieval quality — did we fetch the right chunks? Metrics: recall@k, precision@k, hit-rate, MRR.
  • Generation quality — given the chunks, is the answer faithful (grounded, no hallucination) and relevant (answers the question)? Measured with an LLM-as-judge or frameworks like RAGAS.
Debug the right half
Always separate the two. If retrieval@k is low, no prompt tweak will help — fix chunking/search. If retrieval is good but answers are wrong, fix the prompt/model. Conflating them wastes days.

Interview Q&A

coreWalk me through a RAG pipeline.

Offline: split documents into overlapping chunks, embed each with an embedding model, and store the vectors (plus metadata) in a vector index. Per query: embed the question, retrieve the top-k nearest chunks (ideally hybrid dense + keyword), re-rank them with a cross-encoder to keep the best few, then pass those chunks plus the question to the LLM with an instruction to answer only from the provided context and cite sources. The output is grounded in your data and updates the moment you re-index.

coreRAG or fine-tuning — how do you decide?

RAG for knowledge (facts, private/fresh data, anything needing citations or frequent updates); fine-tuning for behavior (format, tone, a specialized skill). If the answer is “the model doesn’t know our data,” that’s retrieval, not training. They also compose: fine-tune for how to respond, RAG for what to respond with. And per the ladder, try a good prompt first — many “we need to fine-tune” asks are really vague prompts.

trapYour RAG system gives a wrong answer. How do you debug it?

Isolate the stage. First check retrieval: were the correct chunks in the top-k? If not, it’s a retrieval problem — revisit chunk size/overlap, embedding model, or add hybrid search / re-ranking. If the right chunks were retrieved but the answer is still wrong, it’s a generation problem — tighten the prompt (“answer only from context, say ‘not found’ otherwise”), lower temperature, or handle lost-in-the-middle by re-ranking best chunks first. Measuring retrieval and generation separately is the whole game.

seniorWhy hybrid search instead of pure vector similarity?

Dense embeddings capture meaning but are weak on exact tokens — product codes, error IDs, rare names, acronyms. Sparse/BM25 nails those but misses synonyms. Real queries need both, so we run both and fuse the rankings (e.g. reciprocal rank fusion). Pure vector search quietly fails on keyword-heavy queries, which is exactly the kind of bug that shows up only in production.

seniorWhat is “lost in the middle” and how do you mitigate it?

LLMs attend most strongly to the beginning and end of their context, so relevant information placed in the middle of a long context can be effectively ignored. Mitigations: retrieve fewer but higher-quality chunks (don’t stuff 50 in), re-rank so the most relevant are positioned first, and keep the context tight. It’s a strong argument for good retrieval + re-ranking over simply maximizing context length.

trapHow do you keep a RAG system from citing outdated policy after a document is updated?

Attach metadata (version, effective date, source) to every chunk and filter/sort on it — prefer the most recent, deduplicate near-identical chunks, and drop superseded versions from the index on update. If old and new coexist, the model may pick either, so make recency an explicit retrieval constraint rather than hoping the model chooses correctly.

Code: a minimal RAG pipeline

The whole idea in ~40 lines — embed, cosine-search, ground the answer. No vector-DB dependency so the mechanics are visible; in production you’d swap the numpy search for a real index (see Vector DBs & Embeddings).

pythonminimal RAG: embed → retrieve → grounded answer
# Minimal RAG, from scratch, so every stage is visible.
# pip install sentence-transformers numpy openai
from sentence_transformers import SentenceTransformer
import numpy as np
from openai import OpenAI

# ---- INDEX (offline): chunk -> embed -> store -----------------------------
# In reality these come from splitting your documents into overlapping chunks.
chunks = [
    "The Acme Pro plan costs $49/month and includes 10 seats.",
    "Refunds are available within 30 days of purchase.",
    "The Acme Free plan includes 1 seat and community support only.",
    "Enterprise pricing is custom; contact sales for a quote.",
]

embedder = SentenceTransformer("all-MiniLM-L6-v2")     # small, CPU-friendly encoder
chunk_vecs = embedder.encode(chunks, normalize_embeddings=True)  # (n_chunks, dim)

def retrieve(query: str, k: int = 2):
    """Embed the query and return the k most similar chunks by cosine similarity."""
    q = embedder.encode([query], normalize_embeddings=True)[0]     # (dim,)
    # Vectors are normalized, so a dot product IS cosine similarity.
    scores = chunk_vecs @ q                                        # (n_chunks,)
    top = np.argsort(scores)[::-1][:k]                             # indices of best k
    return [chunks[i] for i in top]

# ---- QUERY (online): retrieve -> ground the LLM ---------------------------
client = OpenAI()

def answer(query: str) -> str:
    context = retrieve(query)                                      # the "open book"
    context_block = "\n".join(f"- {c}" for c in context)
    # The instruction is the anti-hallucination guardrail: answer ONLY from context.
    prompt = (
        "Answer the question using ONLY the context below. "
        "If the answer is not in the context, say 'I don't know.'\n\n"
        f"Context:\n{context_block}\n\nQuestion: {query}"
    )
    resp = client.chat.completions.create(
        model="gpt-4o-mini",
        temperature=0,                                            # factual -> deterministic
        messages=[{"role": "user", "content": prompt}],
    )
    return resp.choices[0].message.content

print(answer("How much is the Pro plan?"))     # grounded: "$49/month, 10 seats."
print(answer("Do you offer a free trial?"))    # not in context -> "I don't know."
Connect it to what you know
The indexing phase is a familiar batch/ETL job — read docs, transform (chunk), compute a feature (embedding), load into a store — the kind of thing you’d run on Databricks/Spark. The query phase is a low-latency service call. RAG is 80% data engineering you already know, plus one embedding step and one LLM call.
