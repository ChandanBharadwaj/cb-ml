When to Use ML — the Ladder

The spine of this whole site, and the most valuable instinct you can bring to an interview or a customer: use the cheapest tool that actually works. Don’t reach for a model because it’s exciting — climb only as high as the problem forces you. This page turns that instinct into a repeatable decision process.

The one principle
Can a plain algorithm / rules solve it? → if no, can classical ML? (cheap, CPU, interpretable) → if not enough, deep learning? → only then an LLM? And at every rung: CPU or GPU? what’s the cost & latency?

Decision walker — answer to find your rung

Work a real problem through it. Each answer takes you one step; the result names the rung, the compute, and the rough cost — and why you stopped there.

The four rungs in detail

RungReach here whenCompute · cost
1 · Algorithm / rulesThe rule is stateable & stable (formula, thresholds, lookup, sort)CPU · ~free
2 · Classical MLTabular data, a pattern you can’t hand-write, labels available; want interpretabilityCPU · cheap
3 · Deep learningUnstructured data (images/audio/signals) with lots of labels; classical ML plateausGPU · $$
4 · LLM / GenAILanguage understanding/generation, few labels, need zero/few-shot generalityGPU / API · $/token
Why “cheapest that works” wins
Lower rungs are cheaper to run, faster, easier to explain, simpler to monitor, and less likely to hallucinate. A gradient-boosted tree answering in microseconds on a CPU beats an LLM that needs a GPU and a bag of inference tricks — when it clears the bar. The skill is knowing when it does.

The questions to ask (before any code)

  • Can I state the rule? If the logic is expressible, write it — rung 1. No data, no model, no drift.
  • Is there really a pattern, and enough data to learn it? No pattern or no data → ML won’t help; get data or use rules.
  • Structured or unstructured? Tabular → classical ML first. Text/image/audio → DL or LLM.
  • What does an error cost? High-stakes/regulated → favor interpretable models and lower rungs.
  • Is the cheaper rung good enough? Always baseline the rung below before climbing.
  • CPU or GPU? What’s the latency/cost budget? A GPU-only solution may be a non-starter for the customer — that alone can decide the rung.

The sub-ladder inside “LLM”

Even once you’re at rung 4, keep climbing cheapest-first — this is the decision that comes up most in modern interviews:

Prompt (instructions) → RAG (knowledge) → fine-tune (behavior). Most “we need to fine-tune” problems are solved a rung earlier.

Buy vs build

Orthogonal to the ladder, but the same frugality: for most GenAI features, start by buying — a hosted API model gets you to value fastest. Build/self-host when you have a hard constraint: data can’t leave the customer’s environment, cost at your volume beats the API, you need an on-device/offline model, or you require capabilities a vendor won’t give you. Don’t train from scratch unless you truly must — fine-tune or prompt an existing model first.

Interview Q&A

coreA stakeholder says “let’s use AI for this.” How do you respond?

I start from the problem, not the tool, and climb the ladder. First: can we state the rule? If so, plain logic is cheapest and most reliable. If not, is there a pattern and enough data — and is it tabular (classical ML, likely gradient-boosted trees on a CPU) or unstructured (deep learning / an LLM)? I baseline the cheapest viable rung and only climb when it can’t clear the bar, weighing error cost, interpretability, latency, and whether we even need a GPU. “AI” is an outcome, not a requirement.

trapWhen should you NOT use machine learning at all?

When the rule is explicitly stateable (use code/logic — cheaper, exact, debuggable); when there’s no real pattern or not enough quality data to learn one; when a mistake’s cost demands full determinism/explainability that a model can’t give; or when the operational burden (data pipelines, monitoring, drift, GPUs) outweighs the benefit. ML earns its complexity only when the pattern is real, learnable, and not economically expressible as rules.

coreWhy start with a simple model / baseline?

A simple baseline (rules, logistic regression, gradient-boosted trees) is fast to build, sets the bar the fancy model must beat, is easy to explain and deploy, and often turns out to be good enough — saving a lot of cost and time. It also surfaces data problems early. You only justify a heavier model by showing it meaningfully beats the baseline on a metric that matters, not because it’s more sophisticated.

seniorTabular data with 50k rows — LLM or gradient-boosted trees?

Gradient-boosted trees, almost certainly. On structured/tabular data they typically match or beat deep learning and LLMs, train in seconds on a CPU, are interpretable, and are trivial to deploy and monitor. An LLM here is slower, pricier, harder to make deterministic, and usually less accurate on numeric features. I’d only consider an LLM if the task needed free-text understanding the columns can’t capture — and even then I’d likely use it to create features for the tree.

seniorWithin an LLM solution, how do you decide prompt vs RAG vs fine-tune?

In cost order. Prompt engineering first — it’s free and instant, and fixes most “vague behavior” issues. If the gap is missing knowledge (private/fresh facts, needs citations), that’s RAG. Fine-tune only when you need consistent behavior/format/tone at scale, a specialized skill, or to shrink a big model into a cheaper specialized one — and it doesn’t add facts well, so it’s not the fix for “doesn’t know our data.” They also compose: fine-tune behavior, RAG knowledge.

seniorHow does “must run on CPU / on-device” change your choice?

It can decide the rung outright. A GPU-only solution may be a non-starter at a customer with no GPUs or strict data-residency, so I’d bias toward lower rungs: classical ML runs happily on CPU; if I need a language model, a small quantized SLM can run on-device or on-prem. The compute and deployment constraint is a first-class input to the decision, not an afterthought — sometimes the “best” model is the one that fits where it has to run.

Code: climb the ladder, logging cost/compute

The disciplined workflow in code: try each rung, stop at the first that clears the bar, and record what it costs. This is the habit that keeps solutions cheap.

pythonbaseline-first: rules → GBM → (only if needed) an LLM
# Climb the ladder: try the cheapest rung first, stop as soon as it clears the bar.
TARGET = 0.90            # the accuracy the business needs
# (metrics/evaluation page shows how to choose this threshold honestly)

# --- Rung 1: rules. CPU, ~free, instant, fully explainable. -----------------
def rules_baseline(rows):
    # e.g. flag as urgent if it literally contains a known outage keyword.
    return [("URGENT" if "down" in r["text"].lower() else "NORMAL") for r in rows]

acc = evaluate(rules_baseline, test)          # your eval harness
print(f"rung1 rules      acc={acc:.2f}  compute=CPU  cost=$0")
if acc >= TARGET:
    ship("rules"); raise SystemExit           # stop! no model needed.

# --- Rung 2: classical ML. CPU, cheap, interpretable. -----------------------
from sklearn.ensemble import HistGradientBoostingClassifier
gbm = HistGradientBoostingClassifier().fit(X_train, y_train)   # seconds on a CPU
acc = gbm.score(X_test, y_test)
print(f"rung2 GBM        acc={acc:.2f}  compute=CPU  cost=~$")
if acc >= TARGET:
    ship("gbm"); raise SystemExit             # stop! no GPU, no LLM.

# --- Rung 4: only now reach for an LLM. GPU/API, $/token. -------------------
# (rung 3 deep learning would go here for image/audio problems)
def llm_classifier(rows):
    return [call_llm(r["text"]) for r in rows]   # per-call cost + latency
acc = evaluate(llm_classifier, test)
print(f"rung4 LLM        acc={acc:.2f}  compute=GPU/API  cost=$$$ per call")
ship("llm")   # justified ONLY because the cheaper rungs missed the bar.

# The log tells the story an interviewer/customer wants: you spent the least
# compute and money that met the requirement — and can prove why.
The instinct to internalize
Every problem starts at the bottom of the ladder. You climb only when the rung below provably can’t clear the bar — and you can always say why you stopped where you did. That sentence is what makes you sound senior.
