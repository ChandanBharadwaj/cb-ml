What is Machine Learning?

Before any specific model, one idea has to be crystal clear: machine learning is how a computer learns a task from examples instead of being told the rules by hand. This page builds the vocabulary the rest of the site leans on — the ways models learn, the words for the data, the two phases (training and inference), and the full lifecycle from raw data to a monitored production system.

What machine learning actually is

What it is

Machine Learning (ML) is a way of writing programs by example. Instead of a human spelling out every rule, you show the computer many examples of a task, and an algorithm finds the pattern — a function that maps inputs to outputs — on its own.

Contrast the two styles. In traditional programming you write the rules and the computer applies them: if income > X and age < Y then approve. In machine learning you supply the examples (thousands of past decisions) and the computer writes the rule for you, expressed as a set of numbers called parameters. You bring the data and the goal; the algorithm brings the rule.

That flip is powerful exactly when the rules are too messy to write by hand. Nobody can write an if/else that recognises a cat in a photo or judges the sentiment of a sentence — but given enough labelled examples, a model can learn to. Machine learning is a subset of Artificial Intelligence (AI); deep learning is in turn a subset of machine learning that uses many-layered neural networks.

Bottom line
Reach for ML only when a plain algorithm can’t state the rule. If the rule is stateable and stable, hand-written code is cheaper, faster, and easier to debug — that’s rung 1 of the escalation ladder, and it beats a model every time it applies.

The four ways a model can learn

What it is

A learning paradigm is the kind of feedback a model learns from. There are four you should be able to name and tell apart: supervised, unsupervised, self-supervised, and reinforcement learning.

ParadigmLearns fromAnswers the questionEveryday example
SupervisedLabelled examples (input → known answer)“What is the answer for this input?”Spam vs not-spam, house-price prediction
UnsupervisedUnlabelled data (structure only)“What natural groups or patterns exist?”Customer segmentation, anomaly detection
Self-supervisedData that labels itself (predict a hidden part)“What’s missing / what comes next?”How Large Language Models pre-train (predict the next word)
Reinforcement (RL)Rewards from trial and error“Which action maximises long-run reward?”Game-playing, robot control, tuning chatbots

Supervised learning is the workhorse of business ML: you have a column you want to predict (the label) and history to learn from. Unsupervised learning finds structure when you have no labels — clusters of similar customers, or points that don’t fit any cluster (anomalies). Self-supervised learning is the trick that made modern Generative AI possible: the data provides its own labels (“predict the next token”), so a model can learn from the entire internet without anyone hand-labelling it. Reinforcement learning learns by acting and receiving a reward signal — the same idea behind aligning chatbots to human preferences.

Bottom line
Match the paradigm to the feedback you actually have. Labels → supervised. No labels but you want structure → unsupervised. A giant unlabelled corpus → self-supervised pre-training. A system that acts and gets scored → reinforcement learning.

Features, labels & the data split

What it is

A feature is an input the model reads; a label is the answer it’s trying to predict; and the data split is how you carve your dataset into separate slices for training, tuning, and honest testing.

Picture a spreadsheet. Each row is one example. Most columns are features (age, income, last-login) — the evidence. One column is the label (did-they-churn?) — the answer you want to predict. A supervised model learns the relationship features → label.

The catch: a model that has simply memorised the training rows looks perfect on them and fails on anything new. To catch that, you split the data into three disjoint slices:

  • Training set — the examples the model learns from.
  • Validation set — held-out data you use to tune choices (which model, which hyperparameters) without peeking at the final exam.
  • Test set — locked away until the very end, used once to estimate how the model will do on genuinely unseen data.

The whole point is generalization: performance on data the model never saw during training. A model that scores 99% on the training set but 70% on the test set is overfitting — memorising instead of learning. (The full treatment of over- and under-fitting, the bias–variance trade-off, and data leakage lives in Part 3 → Fitting & Generalization.)

The number-one rookie mistake
Never let information from the test set touch training — not through scaling stats, not through feature selection, not through a duplicated row. That’s data leakage, and it produces a model that dazzles in the notebook and collapses in production.

Training vs inference

What it is

Training is the one-time (or periodic) phase where the model learns its parameters from data. Inference is every time afterward that you use the trained model to make a prediction on a new input.

They have completely different cost profiles, and conflating them is a classic interview slip. Training is expensive and bursty: it chews through the whole dataset many times, often on a Graphics Processing Unit (GPU), and you do it relatively rarely. Inference is cheap-per-call but constant: it runs on every request in production, so its latency and per-call cost are what your users and your bill actually feel.

Training produces the model (learns the parameters); inference consumes it. Different frequency, different cost, different hardware needs.
Bottom line
When someone says “the model is expensive,” ask which phase. Training cost is a one-off capital expense; inference cost is a per-request operating expense that scales with traffic — and it’s usually the one that decides whether a project is viable.

Parameters vs hyperparameters

What it is

A parameter is a value the model learns from data during training. A hyperparameter is a value you choose before training that controls how the learning happens.

The line is “who sets it.” The model sets its parameters — the weights of a neural network, the split thresholds of a tree. A large language model has billions of them (that’s what “7B” means: seven billion learned parameters). Hyperparameters are your dials: the learning rate, the number of trees, the network depth, how long to train. You tune them against the validation set, because they’re choices about the model rather than things the model discovers.

Bottom line
Parameters are learned; hyperparameters are chosen. When you “train” you’re fitting parameters; when you “tune” you’re searching hyperparameters. Keep the two words straight and a lot of ML conversation snaps into focus.

What it is

Interview Q&A

coreIn one sentence, how is machine learning different from traditional programming?

In traditional programming a human writes the rules and the computer applies them; in machine learning you provide examples and the computer infers the rules itself, as learned parameters. You use ML precisely when the rules are too complex or fuzzy to write by hand.

coreSupervised vs unsupervised vs self-supervised — give an example of each.

Supervised learns from labelled pairs (input → answer): e.g. spam classification. Unsupervised finds structure in unlabelled data: e.g. clustering customers into segments. Self-supervised creates labels from the data itself by hiding part of it: e.g. a language model predicting the next token, which is how LLMs pre-train on unlabelled text at internet scale.

trapWhy do we need a separate validation and test set — isn’t one hold-out enough?

Because you make decisions against the validation set (model choice, hyperparameters), you slowly “leak” it into your process — the model becomes indirectly fitted to it. The test set is touched once, at the end, to get an honest estimate of unseen performance. Without that separation you’ll be optimistic about how the model will actually do in production.

coreParameter vs hyperparameter?

A parameter is learned by the model during training (a weight, a tree threshold). A hyperparameter is set by you before training and controls the learning process (learning rate, number of trees, network depth). You fit parameters on the training set and tune hyperparameters against the validation set.

seniorWhere does most of the effort go in a real ML project?

Rarely in the model itself. Sourcing and cleaning data, engineering features, building a trustworthy evaluation, and — once deployed — monitoring for drift and retraining consume the bulk of the time. The lifecycle is a loop (data → features → train → evaluate → deploy → monitor → back to data), and treating modelling as the whole job is a red flag.

trapYour model scores 99% in training but 72% on the test set. What’s happening?

That gap is overfitting — the model memorised the training data instead of learning a pattern that generalises. Fixes include simplifying the model, adding regularization, getting more/represen­tative data, and checking for leakage. The detailed treatment is in Part 3 → Fitting & Generalization.

Code: a whole ML project in 20 lines

Every term on this page — features, labels, the split, training, inference, evaluation — shows up in one tiny scikit-learn script. This is the lifecycle in miniature.

pythonthe ML lifecycle in miniature (scikit-learn)
# A complete supervised-learning project, start to finish, on a CPU in seconds.
# It touches every term from this page: features, labels, split, train, infer, evaluate.
from sklearn.datasets import load_breast_cancer
from sklearn.model_selection import train_test_split
from sklearn.ensemble import RandomForestClassifier
from sklearn.metrics import accuracy_score

# 1) DATA: features X (the evidence) and labels y (the answer we want to predict).
data = load_breast_cancer()
X, y = data.data, data.target          # X: (569, 30) features   y: 0/1 label per row

# 2) SPLIT: keep a test set the model never learns from (honest generalization check).
X_train, X_test, y_train, y_test = train_test_split(
    X, y, test_size=0.2, random_state=0, stratify=y
)

# 3) CHOOSE a model. n_estimators is a HYPERPARAMETER — we set it, the model doesn't.
model = RandomForestClassifier(n_estimators=200, random_state=0)

# 4) TRAIN: the model LEARNS its PARAMETERS (the trees) from the training data.
model.fit(X_train, y_train)

# 5) INFER: use the frozen, trained model to predict on data it has never seen.
preds = model.predict(X_test)

# 6) EVALUATE: how well did it GENERALIZE? (compare predictions to the held-out truth)
print(f"test accuracy: {accuracy_score(y_test, preds):.3f}")
# Next step in a real project (Part 4): deploy this model and MONITOR it in production.
