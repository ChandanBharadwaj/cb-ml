MLOps & Production

A model isn’t done when it trains — it’s done when it runs reliably for months and you notice the day it quietly breaks. Monitoring is consistently the biggest gap in senior candidates, and it’s home turf for you: this is the deployment, observability, and reliability engineering you already do — with model-specific twists.

Drift: the model didn’t change, the world did

Your model is frozen at training time; reality keeps moving. That mismatch is drift, and it silently erodes performance:

  • Data drift (covariate shift) — the input distribution changes. New user segment, a rebranded product name, a sensor recalibrated. Inputs look different than training.
  • Concept drift — the relationship between inputs and the target changes. “Normal” spending patterns shift after a holiday; what counted as fraud last year doesn’t now.
  • Label lag — you often can’t measure accuracy live because ground-truth labels arrive late (did this “churn” prediction come true? ask in 90 days). So you monitor proxies — input distributions and prediction distributions — not just accuracy.
Nobody touched the model, yet accuracy decays as the world drifts. Monitoring catches the crossing; a retraining trigger (or alert) fires before users feel it.

Connect it to what you know
This is dashboards and alerting — your Databricks/PowerBI/cloud-monitoring world. The new idea is monitoring distributions, not just uptime: a drift test on an input feature is a statistical alarm on a data stream, exactly the kind of job you’d run on Spark.

Safe deployment: shadow, canary, A/B

StrategyWhatAnswers
ShadowNew model runs alongside prod on real traffic, but its outputs aren’t served“Does it behave sanely on live data?” — zero user risk

The distinction interviewers want
Shadow/canary answer “is it safe?” (health, errors). A/B answers “is it better?” (a metric moved, with statistical significance). Offline accuracy going up doesn’t prove the business metric will — that’s what the A/B test is for.

Versioning & rollback

Reproducibility in ML means versioning more than code: data, features, model artifacts, hyperparameters, and — for GenAI — prompts too. A model registry (MLflow, etc.) tracks versions and lineage so you can answer “which data + code produced the model serving request X?” and roll back instantly when a new version misbehaves. Treat a prompt change like a code deploy: version it, test it against your eval set, and be able to revert.

Guardrails, PII & responsible AI

The safety layer around a model — especially an LLM that touches customer data:

  • Input/output guardrails — filter prompt injection, block disallowed topics, validate output format/values, catch toxic or unsafe content before it reaches a user.
  • PII handling — detect and redact personal data before it hits a model or a log; respect data-residency (keep it in the customer’s region/VPC) — a real constraint an FDE meets constantly.
  • Bias & fairness — check outcomes across groups; a model can be accurate overall and unfair to a subgroup.
  • Auditability — log inputs, outputs, versions, and decisions so you can explain and defend what the system did (and meet GDPR/EU AI Act obligations).

LLM-specific ops

LLMs add operational concerns beyond classic MLOps: cost & rate limiting (token spend can explode; cap it per user/tenant), prompt/response logging for debugging and eval, continuous evaluation (run your eval set on every prompt/model change), fallbacks (a cheaper model or cached answer when the primary is down or slow), and tracing agent trajectories. Non-determinism makes all of this harder — pin temperature and snapshot model versions where you can.

Interview Q&A

coreData drift vs concept drift — what’s the difference and how do you detect them?

Data drift is a change in the input distribution P(X) — the features look different than at training. Concept drift is a change in the relationship P(y|X) — the same inputs now map to different outcomes. You detect data drift with statistical tests comparing live features to a training baseline (PSI, KL divergence, KS test) and monitoring prediction distributions; concept drift shows up as accuracy decay once labels arrive, or as rising error on a labeled sample. The usual response is an alert and a retraining trigger.

trapHow do you monitor a model when ground-truth labels take 90 days to arrive?

You can’t wait 90 days to notice a problem, so you monitor leading indicators: input feature distributions (data-drift tests) and the model’s output/score distribution, which shift long before delayed labels confirm degradation. You also track business proxies and collect any fast feedback signals. When labels do land, you compute true accuracy on that cohort to confirm. Proxy monitoring now, ground-truth validation later.

coreShadow vs canary vs A/B — when do you use each?

Shadow deployment runs the new model on real traffic without serving its outputs — use it to confirm it behaves and performs sanely with zero user risk. Canary routes a small slice of live traffic to it and ramps up while you watch errors/latency — use it to limit blast radius during rollout. A/B testing splits traffic and compares business KPIs with statistical significance — use it to prove the new model is actually better, not just healthy. They’re complementary stages, not alternatives.

seniorWhat do you version to make an ML system reproducible?

Everything that determines the artifact: code, the training data (or a snapshot/hash), feature definitions, hyperparameters, environment, the model binary, and — for GenAI — prompts and the exact model version/endpoint. A model registry ties these together with lineage so you can reproduce and, critically, roll back a bad release in seconds. Treating prompts as unversioned config is a common and painful mistake.

seniorHow do you keep an LLM system’s cost and safety under control in production?

Cost: cap tokens per request, rate-limit per user/tenant, cache repeated prompts and semantically similar queries, and route easy requests to a cheaper/smaller model. Safety: input and output guardrails (injection filters, allow-lists, toxicity/PII detection and redaction), least-privilege on any tools, human approval for risky actions, and full logging for audit. Plus continuous evaluation on a task-specific set so a prompt or model change can’t silently regress quality.

trapA customer requires that their data never leave their region. How does that shape your design?

Data residency becomes an architectural constraint. I’d favor self-hosted or in-region models (an SLM in their VPC, or a regional endpoint of a provider that contractually keeps data in region), keep embeddings/vector stores and logs in-region, redact PII before anything leaves a trust boundary, and document the data flow for audit. It’s a strong reason to prefer a smaller on-prem model over a public API — exactly the kind of tradeoff an FDE makes at a customer site.

Code: a drift check & a PII guardrail

Two things you’ll actually wire up: a population-stability drift test on a feature, and a PII redactor to run before text hits a model or a log.

pythondata-drift check with Population Stability Index (PSI)
# PSI compares a live feature's distribution to the training baseline.
# Rule of thumb: <0.1 stable, 0.1-0.25 moderate drift, >0.25 significant drift.
import numpy as np

def psi(baseline: np.ndarray, live: np.ndarray, bins: int = 10) -> float:
    # Build bin edges from the TRAINING baseline (quantiles), then reuse them live.
    edges = np.quantile(baseline, np.linspace(0, 1, bins + 1))
    edges[0], edges[-1] = -np.inf, np.inf            # catch out-of-range live values
    base_pct = np.histogram(baseline, edges)[0] / len(baseline)
    live_pct = np.histogram(live,     edges)[0] / len(live)
    eps = 1e-6                                        # avoid log(0) / divide-by-zero
    base_pct, live_pct = base_pct + eps, live_pct + eps
    return float(np.sum((live_pct - base_pct) * np.log(live_pct / base_pct)))

# Baseline captured at training time; 'today' pulled from the live feature store.
baseline = np.random.normal(50, 10, 10_000)          # e.g. avg transaction amount
today    = np.random.normal(58, 12, 2_000)           # distribution has shifted up
score = psi(baseline, today)
print(f"PSI = {score:.3f}", "-> RETRAIN/ALERT" if score > 0.25 else "-> ok")
# Run this per feature on a schedule (a Spark/Databricks job) and alert on breaches.
pythona simple PII redaction guardrail
# Redact obvious PII BEFORE text reaches an LLM, a log, or leaves the trust boundary.
# For production use a real PII detector (e.g. Presidio); regex is the illustrative floor.
import re

PATTERNS = {
    "EMAIL":  r"[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Za-z]{2,}",
    "PHONE":  r"\b(?:\+?\d[\s-]?){10,}\b",
    "SSN":    r"\b\d{3}-\d{2}-\d{4}\b",
    "CARD":   r"\b(?:\d[ -]*?){13,16}\b",
}

def redact(text: str) -> str:
    """Replace detected PII with typed placeholders, e.g. [EMAIL]."""
    for label, pattern in PATTERNS.items():
        text = re.sub(pattern, f"[{label}]", text)
    return text

msg = "Email me at jane.doe@acme.com or call 555-123-4567 about invoice 9921."
print(redact(msg))
# -> "Email me at [EMAIL] or call [PHONE] about invoice 9921."

# so the model can't echo PII back, and keep everything in the customer's region.
The mindset to carry
In production, “accuracy” is table stakes. What earns trust is observability + safe rollout + instant rollback + guardrails — the reliability engineering you already do, applied to a component that also drifts and occasionally makes things up.
