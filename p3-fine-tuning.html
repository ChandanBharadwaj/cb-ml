Fine-tuning & PEFT

The most expensive LLM lever — changing the model’s weights. Done for the right reason it bakes a behavior or style permanently into a smaller, cheaper model. Done for the wrong reason (to add facts) it’s a slow, costly mistake. The single most valuable thing here is knowing when to fine-tune versus RAG versus just prompting.

Where this sits on the ladder
The last LLM lever: prompt → RAG → fine-tune. Reach here only when a good prompt and retrieval aren’t enough — you need consistent behavior/format/style at scale, or to shrink a big model into a cheap specialized one. It costs a training run and needs data, so it’s the priciest option to reach for and to maintain.

Fine-tune vs RAG vs prompt — the decision

This is the question you’ll be asked, and the one that matters on the job. The mental model: prompt changes instructions, RAG changes knowledge, fine-tuning changes behavior.

NeedReach forWhy
Different instructions / format, quick iterationPromptFree, instant, no infra
The model must know our data (facts, fresh, private)RAGInject knowledge at query time; cite & update instantly
Consistent behavior/tone/format at scale; a narrow skill; shrink a modelFine-tuneBakes the behavior into weights; smaller prompt, lower per-call cost
The classic trap
“The model doesn’t know our docs, so let’s fine-tune.” Almost always wrong — that’s a knowledge gap, which is RAG. Fine-tuning teaches how to respond, not what facts to know. And they compose: fine-tune the behavior, RAG the knowledge.

Full fine-tuning vs PEFT

Full fine-tuning updates every weight — maximum flexibility, but you need a copy of the whole model’s worth of gradients/optimizer state in GPU memory (huge), and you get a full new model per task. PEFT (Parameter-Efficient Fine-Tuning) freezes the base model and trains a tiny number of new parameters — usually <1% — getting most of the quality for a fraction of the cost. PEFT is the default in practice, and LoRA is the PEFT everyone means.

LoRA & QLoRA

LoRA (Low-Rank Adaptation) is the key idea to be able to draw. Instead of updating a big weight matrix W (say 4096×4096), freeze it and learn a small low-rank update ΔW = B·A, where A and B are skinny matrices (rank r, e.g. 8–64). You train only A and B — a few million params instead of billions — then add W + BA at inference.

LoRA: freeze W, learn a low-rank ΔW = B·A. The two skinny matrices hold almost all the useful adaptation at a tiny fraction of the parameters.

QLoRA goes further: it quantizes the frozen base model to 4-bit (shrinking its memory ~4×) while training the LoRA adapters in higher precision on top. That’s what lets you fine-tune a big model on a single consumer GPU — a huge deal for cost and for keeping data on-prem.

Why LoRA is a big deal for an FDE
Adapters are tiny (megabytes). You can keep one frozen base model and hot-swap a per-customer LoRA adapter — cheap to train, cheap to store, cheap to serve, and the customer’s data never trains the shared base. That’s a deployment superpower.

Instruction tuning

A raw pretrained model just continues text; it isn’t helpful out of the box. Instruction tuning is supervised fine-tuning (SFT) on many (instruction → good response) pairs, which teaches the model to follow instructions and adopt a helpful assistant style. It’s the first step that turns a base model into something chat-like — before any alignment.

Alignment: RLHF, DPO, GRPO

After SFT, alignment tunes the model toward human preferences — helpful, honest, harmless — and toward the answers people actually prefer.

MethodHowNote
RLHFTrain a reward model from human preference pairs, then RL (PPO) the LLM to maximize itPowerful, but complex & unstable (3 models in play)
DPOOptimize directly on preference pairs — no separate reward model or RL loopSimpler, stable, now a common default
GRPOGroup-relative RL that skips the value model; scores groups of samplesPopular for reasoning models (e.g. DeepSeek-R1)
One-liner
“RLHF aligns via a learned reward model + RL; DPO gets similar results by optimizing preference pairs directly, which is simpler and more stable; GRPO is an RL variant popular for reasoning training. Most teams reach for DPO first now.”

Catastrophic forgetting

When you fine-tune on a narrow dataset, the model can overwrite general capabilities it had — it gets great at your task and worse at everything else. Mitigations: use PEFT/LoRA (the base stays frozen, so general knowledge is preserved), mix in some general data, keep learning rates low, and don’t over-train. It’s a strong argument for LoRA over full fine-tuning when you only have task-specific data.

Interview Q&A

coreExplain LoRA. Why is it so widely used?

LoRA freezes the pretrained weights and learns a low-rank update ΔW = B·A for the target matrices, where A and B are small (rank r). You train only those few million parameters instead of billions, so it needs far less GPU memory and produces tiny, swappable adapter files — while recovering most of full fine-tuning’s quality. It’s popular because it makes fine-tuning cheap, storable, and per-task swappable, and it sidesteps catastrophic forgetting by leaving the base frozen.

coreFine-tuning vs RAG — when each?

RAG for knowledge (facts, private/fresh data, citations, frequent updates); fine-tuning for behavior (consistent format/tone, a specialized skill, or shrinking a big model into a cheaper specialized one). If the gap is “the model doesn’t know our data,” that’s retrieval, not training. They compose — fine-tune how it responds, RAG what it responds with — and per the ladder you try prompting first.

seniorWhat does QLoRA add over LoRA, and what’s the tradeoff?

QLoRA quantizes the frozen base model to 4-bit (using NF4 + double quantization) so it fits in far less memory, then trains LoRA adapters in higher precision on top. The win is being able to fine-tune large models on a single GPU. The tradeoff is a small quality hit from quantizing the base and somewhat slower training due to dequantization, but in practice it’s close to LoRA quality at a fraction of the hardware — which is why it democratized fine-tuning.

seniorRLHF vs DPO — why did DPO catch on?

RLHF trains a separate reward model from human preference pairs and then uses RL (PPO) to optimize the LLM against it — effective but complex and unstable, juggling three models. DPO derives a loss that optimizes the policy directly on the preference pairs, mathematically equivalent to the RLHF objective under common assumptions, with no reward model or RL loop. It’s simpler, more stable, and cheaper to run, so many teams now default to it — reaching for full RLHF/GRPO only when they need the extra control.

trapYou fine-tuned a model and now it’s worse at general tasks. What happened?

Catastrophic forgetting — training on a narrow dataset overwrote general capabilities. Fixes: switch to LoRA/PEFT so the base weights stay frozen; mix general-purpose data into the training set; lower the learning rate and train for fewer steps; and evaluate on a broad held-out set, not just your task, so you catch the regression. It’s a key reason PEFT is preferred when you only have task-specific data.

seniorHow would you serve fine-tuned models for 50 different customers cost-effectively?

Train a LoRA adapter per customer on top of one shared, frozen base model. Adapters are only a few megabytes, so you store 50 of them cheaply and load the relevant one per request (or serve many with adapter-aware batching). You keep a single base model in GPU memory instead of 50 full models, each customer’s data trains only their adapter, and updates are cheap. That’s the canonical LoRA deployment pattern — and a great FDE answer.

Code: LoRA fine-tuning with Hugging Face

The real workflow: load a base model, wrap it with a LoRA config (PEFT), and train only the adapters. Note how few lines turn a frozen model into a trainable one — and how few parameters actually get updated.

pythonLoRA fine-tuning with peft + Trainer
# Parameter-efficient fine-tuning with LoRA. Only the adapters train; base stays frozen.
# pip install transformers peft datasets accelerate
from transformers import AutoModelForCausalLM, AutoTokenizer, TrainingArguments, Trainer
from peft import LoraConfig, get_peft_model
from datasets import load_dataset

base = "Qwen/Qwen2.5-0.5B"                       # small base so this runs on modest hardware
tok = AutoTokenizer.from_pretrained(base)
tok.pad_token = tok.eos_token
model = AutoModelForCausalLM.from_pretrained(base)

# 1) LoRA config: WHERE to inject adapters and how big (rank r).
lora = LoraConfig(
    r=8,                      # rank of the update — bigger = more capacity, more params
    lora_alpha=16,            # scaling factor for the LoRA update
    target_modules=["q_proj", "v_proj"],   # attach to attention query/value projections
    lora_dropout=0.05,
    task_type="CAUSAL_LM",
)

# 2) Wrap the frozen base with LoRA — this is the whole trick.
model = get_peft_model(model, lora)
model.print_trainable_parameters()
# -> trainable params: ~0.5M || all params: ~500M || trainable%: ~0.1%
#    You are training a tiny sliver of the model.

# 3) Tiny instruction-style dataset (in reality: your task-specific pairs).
data = load_dataset("json", data_files="train.jsonl")["train"]
def fmt(ex):
    text = f"### Instruction:\n{ex['instruction']}\n### Response:\n{ex['response']}"
    out = tok(text, truncation=True, max_length=256, padding="max_length")
    out["labels"] = out["input_ids"].copy()      # causal LM: predict the same tokens
    return out
data = data.map(fmt, remove_columns=data.column_names)

# 4) Train — standard Trainer loop, but only the LoRA params receive gradients.
args = TrainingArguments(output_dir="out", per_device_train_batch_size=4,
                         num_train_epochs=3, learning_rate=2e-4, logging_steps=10)
Trainer(model=model, args=args, train_dataset=data).train()

# 5) Save just the ADAPTER (a few MB), not the whole model. Swap it in at inference.
model.save_pretrained("my-lora-adapter")
# Serve: load the base once + this adapter. Keep one adapter per task/customer.
The pattern to remember
Prompt → RAG → fine-tune, and when you do fine-tune, reach for LoRA/QLoRA first: tiny, cheap, swappable, forgetting-resistant. Full fine-tuning is the exception, not the rule.
