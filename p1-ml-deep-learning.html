ML & Deep Learning

The two lowest rungs above plain code. Before anyone reaches for a transformer, they must be able to say why not gradient-boosted trees on a CPU? This page gives you both the classical-ML baselines that still win and the deep-learning machinery — and the judgment to pick between them.

Classical ML: the baselines that still win

“Modern” does not mean “always a neural network.” On a spreadsheet — rows of customers, transactions, sensor readings — a handful of classical models remain the strongest, cheapest, most explainable option. Know them cold; they are your rung-2 default and the thing you must out-argue before spending on a GPU.

The line to remember in an interview
“For tabular data I start with gradient-boosted trees. They beat deep learning on most structured datasets, train on a CPU in seconds, and give me feature importances I can show a stakeholder. I only move to deep learning when the signal is in unstructured data or the trees plateau.”

Why gradient-boosted trees rule tabular data

Boosting builds an ensemble of shallow decision trees, one at a time, where each new tree is trained to correct the residual errors of the trees so far. It’s gradient descent — but in “function space,” adding a tree at each step instead of nudging weights. The result captures messy non-linear interactions between columns without you hand-crafting them, which is exactly what tabular data needs.

Boosting = additive correction. The red error bar shrinks as each shallow tree cleans up the previous ones’ mistakes.

From neuron to network

A single artificial neuron is just weighted sum → add bias → non-linear activation: a = σ(w·x + b). Stack neurons into layers and layers into a network (a Multi-Layer Perceptron), and you get a function that can approximate almost anything — the universal approximation property. The catch: it only becomes useful once you train the weights, and it’s hungry for data and compute.

Signal flows left → right (the forward pass). Every edge is a learned weight; every node applies an activation.

The forward pass

Inference is just matrix multiplications interleaved with activations. For one layer: h = σ(W·x + b), then that h becomes the input to the next layer. Because it’s all matrix math, GPUs — which are enormous parallel multiply-add machines — run it orders of magnitude faster than CPUs. That’s the whole reason deep learning lives on rung 3’s GPU line.

Backprop & gradient descent

Training answers one question: how should I nudge each weight to reduce the loss?

  1. Forward pass — compute the prediction and the loss (how wrong it is).
  2. Backward pass (backprop) — apply the chain rule from the loss back through every layer to get each weight’s gradient (its share of the blame).
  3. Update — step each weight a little in the direction that lowers the loss: w ← w − η·∇w, where η is the learning rate.

Repeat over many mini-batches. Gradient descent is literally “roll downhill on the loss surface.”

The ball (a weight) rolls down the loss surface toward the minimum. The learning rate sets its step size: too big and it overshoots; too small and it crawls.

Activations, losses & optimizers

The three knobs you’ll be asked about:

  • Activation — the non-linearity that lets networks model curves, not just lines. ReLU (max(0,x)) is the default hidden-layer choice; GELU/SiLU in transformers; softmax on the output for multi-class probabilities.
  • Loss — what “wrong” means. Cross-entropy for classification, MSE for regression. The loss is what backprop differentiates.
  • Optimizer — how you apply gradients. Plain SGD works; Adam/AdamW adapt the step size per-parameter and converge faster — the modern default.
Vanishing/exploding gradients
Deep stacks can lose or blow up the gradient signal on the way back. The fixes you should name: ReLU-family activations, residual connections, batch/layer normalization, and careful weight initialization.

Ladder rule
Don’t skip rung 2. A gradient-boosted tree baseline is cheap insurance: if it already hits the target, you just saved a GPU budget and a week. Climb to deep learning only when the baseline can’t reach the bar or the signal genuinely lives in unstructured data.

Interview Q&A

coreWhen would you not use deep learning?

For tabular/structured data with modest volume, when I need interpretability, or when a CPU budget matters — which covers a huge share of real business problems. Gradient-boosted trees usually match or beat a neural net there, train in seconds, and are explainable. Deep learning earns its cost on unstructured data (images, audio, language) or when classical models plateau.

coreExplain backpropagation in one breath.

It’s the chain rule applied backwards through the network. After a forward pass computes the loss, backprop propagates the derivative of the loss with respect to each weight from the output layer back to the input, reusing intermediate results. Those gradients tell the optimizer which direction and how much to nudge every weight.

trapWhy do we need a non-linear activation at all?

Without one, stacking layers is pointless: a composition of linear maps is still a single linear map, so a 50-layer network would have the exact same expressive power as one linear layer. The non-linearity (ReLU, GELU…) is what lets the network model curved decision boundaries and complex functions.

seniorSGD vs Adam — when does the choice matter?

Adam adapts a per-parameter learning rate from running estimates of the gradient’s mean and variance, so it converges fast with little tuning — the safe default, especially for transformers (as AdamW). Well-tuned SGD with momentum can generalize slightly better on some vision tasks and uses less memory. In practice: reach for AdamW first; consider SGD if you’re squeezing out the last bit of generalization with a tuning budget.

trapYour gradient-boosted model gets 92% and a neural net gets 92.3% on the same tabular data. Which ship?

The gradient-boosted model. A 0.3% gain doesn’t justify the extra cost: the tree trains on a CPU in seconds, is interpretable, easier to deploy and monitor, and has no GPU dependency. I’d only prefer the net if that margin were both large and business-critical. This is the ladder in action — pick the cheapest rung that clears the bar.

seniorWhat is the universal approximation theorem, and why doesn’t it mean “always use a neural net”?

It says a sufficiently wide network can approximate any continuous function to arbitrary precision. But it’s an existence result — it says nothing about how much data, compute, or tuning you need to actually find those weights, or whether the result generalizes. On tabular data a boosted tree often reaches the same accuracy with a fraction of the cost, so “can represent it” ≠ “is the right tool.”

Code: GBM baseline vs PyTorch MLP

The honest way to make the rung-2-vs-rung-3 decision is to run the baseline first. Here’s the classical model you should almost always try before touching a GPU:

pythonrung 2 — gradient-boosted trees (scikit-learn)
# Rung 2: the tabular baseline you run FIRST. Trains on a CPU in seconds.
# (HistGradientBoosting is sklearn's built-in LightGBM-style booster; for the
#  real thing use xgboost.XGBClassifier or lightgbm.LGBMClassifier — same idea.)
from sklearn.datasets import make_classification
from sklearn.ensemble import HistGradientBoostingClassifier
from sklearn.model_selection import train_test_split
from sklearn.metrics import accuracy_score, roc_auc_score

# A stand-in tabular dataset: 20 features, some informative, some noise.
X, y = make_classification(n_samples=5000, n_features=20, n_informative=8, random_state=0)
X_tr, X_te, y_tr, y_te = train_test_split(X, y, test_size=0.2, random_state=0)

# One line to a strong model — no scaling, no GPU, handles feature interactions.
clf = HistGradientBoostingClassifier(max_iter=200, learning_rate=0.08, max_depth=None)
clf.fit(X_tr, y_tr)                                   # trains in ~1 second on a CPU

pred  = clf.predict(X_te)
proba = clf.predict_proba(X_te)[:, 1]                 # P(class = 1), for AUC/threshold tuning
print(f"accuracy: {accuracy_score(y_te, pred):.3f}")
print(f"ROC-AUC : {roc_auc_score(y_te, proba):.3f}")  # threshold-independent quality
# This number is your bar. Only climb to deep learning if you must beat it.

And here is the rung-3 version — a small neural net in PyTorch, with the full training loop spelled out so you can see forward pass, loss, backprop, and the optimizer step:

pythonrung 3 — an MLP with a hand-written training loop (PyTorch)
# Rung 3: the same tabular task with a neural net, so you can SEE every training step.
# For plain tabular data this is usually overkill — but the loop below is the exact
# pattern behind every deep model, including transformers.
import torch
import torch.nn as nn

# Reuse X_tr/y_tr/X_te/y_te from above; convert to tensors.
Xtr = torch.tensor(X_tr, dtype=torch.float32)
ytr = torch.tensor(y_tr, dtype=torch.float32).unsqueeze(1)   # shape (N, 1)
Xte = torch.tensor(X_te, dtype=torch.float32)

# 1) Define the network: input(20) -> hidden(64) -> ReLU -> output(1).
model = nn.Sequential(
    nn.Linear(20, 64),   # weighted sum + bias  (W·x + b)
    nn.ReLU(),           # non-linearity — without this, layers collapse to one linear map
    nn.Linear(64, 1),    # single logit output
)

# 2) Loss + optimizer. BCEWithLogits = sigmoid + binary cross-entropy in one stable op.
loss_fn   = nn.BCEWithLogitsLoss()
optimizer = torch.optim.AdamW(model.parameters(), lr=1e-3)   # adaptive, the modern default

# 3) The training loop — forward, loss, backward (backprop), step. Repeat.
for epoch in range(200):
    optimizer.zero_grad()          # clear gradients from the previous step
    logits = model(Xtr)            # FORWARD pass: predictions
    loss   = loss_fn(logits, ytr)  # how wrong are we?
    loss.backward()                # BACKPROP: chain-rule gradients for every weight
    optimizer.step()               # UPDATE: w <- w - lr * grad
    if epoch % 50 == 0:
        print(f"epoch {epoch:3d}  loss {loss.item():.4f}")

# 4) Evaluate (no gradients needed at inference).
with torch.no_grad():
    prob = torch.sigmoid(model(Xte)).squeeze()     # logits -> probabilities
    acc  = ((prob > 0.5).int() == torch.tensor(y_te)).float().mean()
print(f"MLP accuracy: {acc:.3f}")
# Compare against the GBM's number. On tabular data they'll be close — which is
# exactly why the cheap CPU model usually wins the deployment decision.
The point of showing both
Same task, two rungs. The GBM is one line and a CPU second; the MLP is a training loop and (at scale) a GPU. On tabular data their accuracy is usually within a hair — so the decision is made on cost, latency, and explainability, not accuracy. That instinct is what the ladder trains.
