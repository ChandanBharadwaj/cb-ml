Part 2 · Building with LLMs

Agents & Tool Use

An LLM on its own can only produce text. An agent wraps it in a loop and gives it tools — search, a database, an API, code execution — so it can take actions, observe the results, and decide what to do next. This is the frontier of “building with LLMs,” and where an FDE spends real time: turning a chat model into something that gets work done.

Where this sits on the ladder
The top of rung 4 — the most powerful and the most expensive. Agents make many LLM calls per task (each loop step is a call), so cost, latency, and failure risk multiply. Use one only when the task genuinely needs multi-step reasoning + tools. For a single known step, a plain LLM call or a workflow beats an agent.
many LLM calls highest $ / latency multi-step + tools

The agent loop (ReAct)

The dominant pattern is ReAct — Reason + Act. The model alternates between thinking, acting (calling a tool), and observing the result, repeating until it has enough to answer. That loop is the whole trick: tools give it fresh, real-world information, and the loop lets it chain steps.

Think → Act → Observe, repeat. Each Act is a tool call; each Observe feeds the result back so the next Think is better informed. The model itself decides when it’s done.

Tool use / function calling

A tool is just a function you expose to the model with a name, a description, and a JSON schema of arguments. The model doesn’t run code — it emits a structured request (“call get_weather(city='Paris')”). Your code executes it and feeds the result back. This is the safety boundary and the source of the agent’s power.

Critical mental model
The model only proposes a call; your code decides whether and how to run it. Never wire a raw model request straight into a dangerous action (delete, pay, email) without validation and permissions — this is exactly where prompt injection turns into real damage. Least privilege on every tool.

Memory

The model is stateless between calls, so agents bolt on memory:

Short-termThe current conversation / scratchpadIn the context window (bounded)

EpisodicRecords of past tasks/actions to learn fromA log/store, summarized back into context
Long-term memory is RAG
“How does an agent remember across sessions?” — you retrieve relevant past facts from a vector store and inject them into the context. Memory and retrieval are the same machinery; that connection is a strong interview answer.

Patterns & multi-agent

  • ReAct — interleave reasoning and tool calls (the default single agent).
  • Plan-and-execute — make a full plan up front, then execute steps; cheaper and more controllable for well-structured tasks.
  • Reflection — the agent critiques its own output and retries; improves quality at the cost of more calls.
  • Multi-agent — specialized agents (planner, coder, reviewer) coordinated by an orchestrator. Powerful for complex work, but more moving parts, cost, and failure surface.
FDE judgment
Multi-agent looks impressive and is often overkill. Every added agent multiplies latency, cost, and the chance of a derailed loop. Reach for the simplest thing that works: one good agent with the right tools beats an orchestra you can’t debug.

MCP — the tool standard

The Model Context Protocol (MCP) is an open standard for connecting models to tools and data sources through a common interface — think “USB-C for AI tools.” Instead of hand-writing a bespoke integration per model and per tool, a tool exposes an MCP server once and any MCP-aware client can use it. For an FDE wiring an LLM into a customer’s systems, that’s a big deal: standardized, reusable connectors instead of one-off glue.

Evaluation & guardrails

Agents are hard to evaluate and operate — errors compound across steps, and a loop can run away. What to put in place:

  • Trajectory eval — judge not just the final answer but whether each step (tool choice, arguments) was correct. A right answer via a lucky wrong path is fragile.
  • Guardrails — max-step / max-cost limits to stop runaway loops, tool allow-lists, output validation, human-in-the-loop approval for risky actions.
  • Observability — trace every think/act/observe step (LangSmith, etc.); you cannot debug an agent you can’t see.
  • Least privilege — scope each tool’s permissions tightly; assume the model can be manipulated (prompt injection) and design so the blast radius is small.

Interview Q&A

coreWhat is an LLM agent, and how is it different from a plain LLM call?

A plain call maps a prompt to text in one shot. An agent puts the LLM in a loop with tools: it reasons, chooses an action (a tool call), observes the result, and repeats until it can answer — the ReAct pattern. That lets it fetch live data, take multi-step actions, and self-correct. The cost is that it makes many calls per task, so it’s slower, pricier, and harder to make reliable — use it only when the task truly needs multi-step tool use.

coreHow does function calling actually work?

You give the model tool definitions (name, description, JSON argument schema). When it decides a tool is needed, it returns a structured call — the tool name and arguments — instead of prose. Your application executes that function, appends the result to the conversation, and calls the model again so it can use the result. The model never runs code itself; it only requests calls, and your code stays in control of execution.

seniorHow do you give an agent long-term memory?

Short-term memory is just the context window (the running conversation). For long-term memory you store facts, summaries, and past interactions in a vector database and retrieve the relevant ones into the prompt when needed — i.e. it’s RAG applied to the agent’s own history. Episodic memory (records of past tasks) works the same way. The key point: the model is stateless, so “memory” is always external storage plus retrieval, not something inside the model.

trapWhen would you NOT use an agent?

When the task is a fixed, known sequence of steps — then a deterministic workflow (or a single structured LLM call) is cheaper, faster, and far more reliable than letting a model decide the path each time. Agents shine when the steps are dynamic and unknown up front. Reaching for a multi-agent system when a simple pipeline would do is a common over-engineering trap: more agents means more cost, latency, and ways to fail.

seniorHow do you keep an agent safe and prevent runaway loops?

Layered controls: hard limits on steps and total cost/tokens per task to stop infinite loops; least-privilege, allow-listed tools so the model can’t take dangerous actions; validation of every tool call’s arguments before execution; human-in-the-loop approval for irreversible operations (payments, deletes, external emails); and full tracing/observability so you can see and debug the trajectory. Underlying principle: treat the model as untrusted and keep execution authority in your code.

coreWhat is MCP and why does it matter?

The Model Context Protocol is an open standard for exposing tools and data sources to LLMs through a uniform interface — a tool implements an MCP server once and any MCP-compatible client can use it, instead of building custom integrations per model/tool. It matters because it makes tool integrations reusable and portable, which is exactly what you want when connecting an LLM to a customer’s existing systems.

Code: a tool-calling agent loop

The ReAct loop in ~40 lines: define a tool, let the model request it, execute it, feed the result back, repeat until it answers. Every real agent framework (LangGraph, etc.) is a more elaborate version of exactly this.

pythona minimal ReAct tool-calling loop
# A minimal agent: the model can call a tool, we run it, feed the result back, repeat.
# pip install openai
from openai import OpenAI
import json

client = OpenAI()

# 1) The actual tool — plain Python your code controls.
def get_weather(city: str) -> str:
    fake = {"Paris": "18C, cloudy", "Tokyo": "26C, clear"}
    return fake.get(city, "unknown")

# 2) Describe the tool to the model as a JSON schema (name + args it may request).
tools = [{
    "type": "function",
    "function": {
        "name": "get_weather",
        "description": "Get current weather for a city.",
        "parameters": {
            "type": "object",
            "properties": {"city": {"type": "string"}},
            "required": ["city"],
        },
    },
}]

messages = [{"role": "user", "content": "Should I take an umbrella in Paris?"}]

# 3) THE AGENT LOOP: think -> (maybe) act -> observe -> repeat, capped for safety.
for step in range(5):                       # max-steps guard against runaway loops
    resp = client.chat.completions.create(
        model="gpt-4o-mini", messages=messages, tools=tools,
    )
    msg = resp.choices[0].message
    messages.append(msg)                    # keep the running conversation (memory)

    if not msg.tool_calls:                  # no tool requested -> the model is done
        print("ANSWER:", msg.content)
        break

    # 4) The model ACTED: execute each requested tool and OBSERVE the result.
    for call in msg.tool_calls:
        args = json.loads(call.function.arguments)      # model-proposed arguments
        # NOTE: in production, VALIDATE args and permissions before executing.
        result = get_weather(**args)                    # your code runs the tool
        messages.append({                               # feed the observation back
            "role": "tool",
            "tool_call_id": call.id,
            "content": result,
        })
    # loop continues: the model now sees the tool result and reasons again.
The pattern to remember
An agent = LLM + tools + a loop + memory + guardrails. Frameworks add ergonomics, but if you can write this loop and explain where validation and step-limits go, you understand agents better than most.
